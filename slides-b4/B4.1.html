// B4.1.1 · Foundation

What is an Abstract Data Type?

An ADT is a high-level description of a data structure — it defines what operations can be performed and what behaviour to expect, without specifying how those operations are implemented.

What ADTs Define
Data: what kind of values are stored
Operations: what actions can be performed (insert, delete, search, etc.)
Behaviour: what the result of each operation should be

ADTs are the interface. The underlying data structure (array, nodes, hash table) is the implementation.
Why ADTs Matter
Abstraction: users of the ADT don't need to know how it works internally
Modularity: swap the implementation without changing code that uses the ADT
Correctness: the defined behaviour is a contract — implementations must honour it

Already seen in B2: Stacks (LIFO) and Queues (FIFO) are ADTs — you learned their operations (push, pop, enqueue, dequeue) without worrying about whether they use arrays or linked lists internally.

Linked Lists — Nodes and Pointers

A linked list is a sequence of nodes, where each node stores a data value and a pointer (reference) to the next node in the list.

SINGLY LINKED LIST — DIAGRAM
HEAD →
10
→
25
→
47
→
63
null
Key Properties
Dynamic size: grows and shrinks at runtime — no fixed capacity like an array
Non-contiguous memory: nodes can be stored anywhere in memory, linked by pointers
Sequential access: to reach node N, you must traverse from the head through N-1 nodes
Operations
Insertion: add node at head, tail, or middle — update pointers
Deletion: remove a node — update pointers to bypass it
Traversal: visit every node from head to tail
Search: scan sequentially until target found

IB Requirement: Be able to sketch insertion, deletion, and traversal diagrammatically — showing nodes and how pointer updates work step by step.

// B4.1.2 · Linked List Variants

Three Types of Linked Lists

Singly
Singly Linked List: each node has one pointer — to the next node. Can only traverse forward.
HEAD → [10|→] → [25|→] → [47|null]
Use when: forward-only traversal is sufficient; memory is limited.
Doubly
Doubly Linked List: each node has two pointers — to the next and previous node. Can traverse both directions.
null ← [10|↔] ↔ [25|↔] ↔ [47|→] → null
Use when: bidirectional traversal needed (e.g. browser history back/forward).
Circular
Circular Linked List: the last node's pointer points back to the head instead of null — forming a loop.
HEAD → [10|→] → [25|→] → [47|→] → HEAD
Use when: continuous cycling is needed (e.g. round-robin scheduling, playlist looping).
Linked List vs Array
Array: fixed size, fast random access O(1), slow insertion/deletion (must shift)
Linked List: dynamic size, slow random access O(n), fast insertion/deletion O(1) once position is known
Linked List Disadvantages
No random access — must traverse from head
Extra memory for pointer storage
Not cache-friendly — nodes scattered in memory
// B4.1.3 · Linked List Operations

Insertion & Deletion Diagrammatically

INSERTION AT HEAD
Before: HEAD → [A] → [B] → [C]
New node [X]
Step 1: X.next = HEAD (point X to A)
Step 2: HEAD = X (move HEAD to X)
After: HEAD → [X] → [A] → [B] → [C]
INSERTION IN MIDDLE (after node B)
Before: ... → [B] → [C] → ...
New node [X]
Step 1: X.next = B.next (point X to C)
Step 2: B.next = X (point B to X)
After: ... → [B] → [X] → [C] → ...
DELETION (remove node B)
Before: [A] → [B] → [C]
Step 1: A.next = B.next (point A to C)
Step 2: B is now unreferenced
After: [A] → [C]
Note: memory for B is freed
TRAVERSAL (search for value 25)
current = HEAD
while current ≠ null:
  if current.data == 25: FOUND
  current = current.next
→ visit each node in order

Key principle: All linked list operations depend on correctly updating pointers. Updating them in the wrong order causes nodes to be lost or loops to form — always set the new node's pointer before breaking existing links.

// B4.1.4 · Binary Search Trees

Binary Search Trees (BSTs)

A BST is a tree structure where each node has at most two children, and every node satisfies the BST property.

BST Property (Invariant)
For any node N:
  • All values in N's LEFT subtree are less than N
  • All values in N's RIGHT subtree are greater than N
EXAMPLE BST (insert: 50, 30, 70, 20, 40)
        50
      /     \
    30      70
   /   \
  20   40
Why BSTs are Useful
Fast search: O(log n) on a balanced tree
At each node, eliminate half the remaining possibilities — like binary search on sorted arrays, but built into the structure

Also supports: in-order traversal produces sorted output
// B4.1.4 · BST Operations

BST: Insert, Search & Traverse

Insert (value: 35)
Start at root (50)
35 < 50 → go Left (30)
35 > 30 → go Right (40)
35 < 40 → go Left (empty)
Insert 35 here ✓
Search (value: 40)
Start at root (50)
40 < 50 → go Left (30)
40 > 30 → go Right (40)
40 == 40 → FOUND ✓
Steps: O(log n) best case
In-Order Traversal
Visit: Left → Root → Right
Result: 20, 30, 40, 50, 70
(sorted ascending order!)

Pre-order: Root→L→R
Post-order: L→R→Root
Deletion — Three Cases
Leaf node (no children): simply remove the node
One child: replace the node with its child — update parent's pointer
Two children: find the in-order successor (smallest value in right subtree), copy its value here, then delete the successor node

Worst case — Unbalanced BST: If values are inserted in sorted order (1, 2, 3, 4, 5…), the BST degenerates into a linked list — O(n) search. Balanced BSTs (AVL, Red-Black) fix this, but are beyond IB scope.

// B4.1.5 · Sets

Sets — Unique, Unordered Collections

A set is a collection of unique elements with no duplicates and no defined order. Based on mathematical set theory.

Fundamental Properties
Uniqueness: each element appears at most once
Unordered: no guaranteed sequence — {1, 2, 3} and {3, 1, 2} are the same set
Membership: O(1) average lookup — "is X in this set?"
Mutable: elements can be added or removed
Set Operations
Union (A ∪ B): all elements from both A and B
Intersection (A ∩ B): only elements in both A and B
Difference (A \ B): elements in A but not in B
Subset: every element of A is also in B (A ⊆ B)
Superset: B ⊆ A (A contains all of B)
// Python examples
A = {1, 2, 3, 4}    B = {3, 4, 5, 6}
A | B  → {1, 2, 3, 4, 5, 6}     # Union
A & B  → {3, 4}               # Intersection
A - B  → {1, 2}               # Difference
5 in A  → False              # Membership check

Java equivalents: HashSet implements a set. Add → add(), Remove → remove(), Contains → contains(), Union → addAll().

// B4.1.6 · Hash Tables

Hash Tables — Fast Key-Value Lookup

A hash table stores data as key-value pairs, using a hash function to compute a table index directly from the key — enabling O(1) average-case lookup.

How it Works
1. Given a key (e.g. "alice"), apply the hash function
2. Hash function computes an integer index (e.g. 4)
3. Store / retrieve the value at that index in an array
4. Lookup: hash the key again → go directly to index → O(1)
Hash Function Properties
Deterministic: same key always produces the same index
Uniform distribution: ideally spreads keys evenly across the table
Fast to compute: should run in O(1)
// Conceptual example
hash("alice") → index 4
hash("bob") → index 1
hash("carol") → index 7

Table:
[0]: empty
[1]: "bob" → 92
[2]: empty
[3]: empty
[4]: "alice" → 88
[5–6]: empty
[7]: "carol" → 74
// B4.1.6 · Hash Tables

Collision Resolution

A collision occurs when two different keys produce the same hash index. Since table slots can hold one value, a strategy is needed to handle this.

Chaining (Separate Chaining)
Each slot holds a linked list of all key-value pairs that hash to that index.

Collision → append to the list at that slot.
Lookup → hash key → scan the list at that index.

Advantage: simple; handles many collisions gracefully.
Disadvantage: extra memory for pointers; long lists degrade performance.
Open Addressing (Linear Probing)
If slot is occupied, try the next slot (index + 1), then + 2, etc., until empty slot found.

Lookup → hash key → check slot; if not match, probe forward.

Advantage: all data stays in the array; cache-friendly.
Disadvantage: clustering — consecutive occupied slots slow probing.
Load Factor
Load factor = (number of stored entries) / (total table capacity)
A high load factor (close to 1.0) increases collision probability and degrades performance toward O(n).
When load factor exceeds a threshold (typically 0.75), the table is rehashed — copied into a larger array with new hash positions.

Java / Python: HashMap and dict use chaining with automatic rehashing. HashSet and Python set are implemented as hash tables that store only keys (no values).

// Summary · All B4 ADTs Compared

ADT Quick Reference

ADT Order Access Method Insert Delete Search Best For
Singly Linked List Sequential Traversal (head→tail) O(1) at head O(n) find, O(1) remove O(n) Frequent head insertions; dynamic size
Doubly Linked List Sequential (bi-dir) Traversal (either dir) O(1) at head/tail O(n) find, O(1) remove O(n) Browser history, undo/redo
Circular Linked List Circular Traversal (loops) O(1) at head O(n) O(n) Round-robin, playlist looping
BST (balanced) Sorted Key comparison O(log n) O(log n) O(log n) Sorted data storage; range queries
Set (HashSet) Unordered Hash function O(1) avg O(1) avg O(1) avg Membership testing; removing duplicates
Hash Table (HashMap) Unordered Hash function (by key) O(1) avg O(1) avg O(1) avg by key Key-value lookups; caching; indexing

Choosing an ADT: Need fast search by key? → Hash table. Need sorted data? → BST. Need dynamic insertion at the front? → Linked list. Need unique elements? → Set. Need ordered LIFO/FIFO? → Stack/Queue (B2).

// Real-World Context

ADTs in Practice

Linked Lists
Browser history: doubly linked list — back/forward navigation
Music playlists: circular — loop through songs
OS memory allocation: free memory blocks tracked in a linked list
Image viewer: next/previous image navigation
BSTs
File system directories: hierarchical sorted storage
Database indexes: B-trees (generalization of BST) for fast record lookup
Spell checkers: dictionary stored as BST for fast word search
Autocomplete: prefix-based BST (trie) for suggestions
Sets
Email deduplication: track which emails have been seen
Search engines: track which URLs have been crawled
Social networks: find mutual friends (intersection of friend sets)
Gaming: track which achievements a player has unlocked
Hash Tables
Database caching: store recent query results for instant access
Symbol tables: compilers store variable names → memory addresses
DNS lookup: domain name → IP address mapping
Password storage: store hashed passwords (not the passwords themselves)
