// B3.2.1 · Inheritance

What is Inheritance?

Inheritance allows a class (child/subclass) to acquire the properties and methods of another class (parent/superclass), promoting code reuse and hierarchical organization.

The "is-a" Relationship
A Customer is a Person. An Employee is a Person. Both inherit name, phone, email, and address from Person — and then add their own specialized fields.
Why It Matters
Write once, reuse everywhere. Common code lives in the parent class. Subclasses only define what makes them different. Changes to shared behavior only need to be made in one place.
Person
- name: str
- phone: str
- email: str
- address: str
+ __str__(): str
△
inherits
Customer
- membership: int
- points: int
- signup: date
+ __str__(): str
Employee
- salary: float
- bankaccount: str
- supervisor: Person
+ __str__(): str
// B3.2.1 · Inheritance in Code

Inheritance — Python

class Person: def __init__(self, name, phone, email, address): self.name = name self.phone = phone self.email = email self.address = address def __str__(self): return "Person: " + self.name # Customer INHERITS Person using parentheses class Customer(Person): def __init__(self, name, phone, email, address, membership, points, signup): # MUST call parent constructor first super().__init__(name, phone, email, address) self.membership = membership self.points = points self.signup = signup def __str__(self): # overrides parent's __str__ return "Customer: " + self.name + " has " + str(self.points) + " points."
①
class Customer(Person) — the parentheses declare the parent class
②
super().__init__(...) — must be the first line in the child constructor to initialize inherited attributes
③
Child class then adds its own unique attributes after calling super()
// B3.2.1 · Access Modifiers in Inheritance

Access Modifiers

Access modifiers control which members of a parent class are accessible in subclasses. This becomes critical when designing class hierarchies.

private
Not inherited. Only accessible within the class that declares it. Subclasses cannot directly access private members — they must use getters/setters.
public
Fully inherited. Accessible everywhere — inside the class, in subclasses, and from outside code. No restrictions.
protected
Inherited but not public. Accessible within the class and all subclasses, but not from outside code. The "middle ground" — common in Java inheritance hierarchies.

Python note: Python uses naming conventions rather than keywords. _name (single underscore) signals "protected" by convention; __name (double underscore) triggers name-mangling and behaves like private. Java uses explicit private, protected, and public keywords.

// B3.2.2 · Polymorphism

What is Polymorphism?

Polymorphism means "many forms". Related objects can respond to the same method call in different ways, depending on their actual class.

Dynamic (Runtime) Polymorphism
Achieved through method overriding. A subclass provides a different implementation of a method that already exists in the parent. The correct version is selected at runtime.

Example: Customer and Employee both override __str__() from Person.
Static (Compile-time) Polymorphism
Achieved through method overloading — same method name but different parameters. Resolved at compile time. More common in Java; Python uses default arguments to simulate this.

Example: add(int a, int b) vs add(double a, double b)

Key insight: Overriding = subclass redefines a parent's method with the same name and signature. Overloading = same class has multiple methods with the same name but different parameters. These are different concepts.

// B3.2.2 · Method Overriding

Overriding — in Practice

When print(obj) is called, Python calls __str__(). Each class has its own version — the correct one runs automatically.

# Main code using polymorphism p = Person("Jordan McFly", "555-1234", "jordan@example.com", "1885 Brown Estate") c = Customer("Skyler Serenity", "555-2345", "skyler@example.com", "1701 Asimov Plaza", 1, 0, "2024-01-15") e = Employee("Avery Shephard", "555-3456", "avery@example.com", "1955 Lone Pine Mall", 75000, "123-456-888", p) print(p) # → Person: Jordan McFly print(c) # → Customer: Skyler Serenity has 0 points. print(e) # → Employee: Avery Shephard earns $75000
①
Each object's actual class determines which __str__() runs — not the variable type
②
If a subclass does not override the method, it falls back to the parent's version
③
Real-world benefit: GUI toolkits (password, date, file inputs) share a base TextInput, each overriding only what differs
// B3.2.3 · Abstraction

Abstract Classes

An abstract class is a template that cannot be instantiated directly. It defines methods that all subclasses must implement — enforcing a consistent interface.

When to Use Abstract Classes
When all subclasses must have certain methods, but the implementation differs for each subclass. The abstract class provides the contract — subclasses provide the implementation.

Example: All 2D shapes must have getArea() and getPerimeter(), but rectangles and circles calculate them differently.
Key Rules
Cannot instantiate an abstract class directly
Must override all abstract methods in subclasses
Can contain both abstract and concrete (implemented) methods
Python: use ABC and @abstractmethod
Java: use abstract class and abstract methods

Benefit: Any code that works with a Shape object is guaranteed that getArea() and getPerimeter() exist — even if it doesn't know whether the shape is a Circle, Rectangle, or Triangle.

// B3.2.3 · Abstract Class in Python

Abstract Class — Code

from abc import ABC, abstractmethod import math class Shape(ABC): # ABC = Abstract Base Class @abstractmethod def get_area(self): ... @abstractmethod def get_perimeter(self): ... class Rectangle(Shape): def __init__(self, width, height): self.width = width self.height = height def get_area(self): return self.width * self.height def get_perimeter(self): return 2 * (self.width + self.height) class Circle(Shape): def __init__(self, radius): self.radius = radius def get_area(self): return math.pi * self.radius ** 2 def get_perimeter(self): return 2 * math.pi * self.radius # Polymorphism + Abstraction working together shapes = [Rectangle(10, 4), Circle(5)] for s in shapes: print(s.get_area(), s.get_perimeter())

Common error: Trying to instantiate an abstract class — s = Shape() — raises a TypeError. Abstract classes exist only to be extended, not instantiated.

// B3.2.4 · Composition & Aggregation

Composition vs Aggregation

Both describe "has-a" relationships between objects — the key difference is lifecycle dependency.

Composition — Strong "has-a"
The contained objects cannot exist independently of the container. If the container is destroyed, the parts are destroyed too — a "death relationship".

Car → Engine (engine doesn't function outside the car)
House → Room (rooms cease to exist if house is demolished)
Human → Heart (heart cannot live outside the body)
Aggregation — Weak "has-a"
The contained objects can exist independently. The container references the object but doesn't own its lifecycle.

University → Student (student continues after university closes)
Library → Books (books exist independently of the library)
Playlist → Song (song exists outside the playlist)

UML notation: Composition uses a filled diamond (◆) on the container end. Aggregation uses an open diamond (◇). Both point from container to contained class.

// B3.2.4 · Composition in Code

Composition — Example

class Engine: def __init__(self, horsepower): self.horsepower = horsepower def start(self): return f"Engine ({self.horsepower}hp) started" # Car COMPOSES Engine — Car creates and owns the Engine class Car: def __init__(self, make, horsepower): self.make = make self.engine = Engine(horsepower) # Engine is created inside Car def drive(self): return f"{self.make}: " + self.engine.start() my_car = Car("Tesla", 300) print(my_car.drive()) # Tesla: Engine (300hp) started # When my_car is deleted, its Engine is also gone
①
Composition: Engine is created inside the Car constructor — the Car owns it
②
Aggregation: Would pass an already-existing object in — e.g. Car(make, existing_engine) — the Engine can exist independently
③
Both let one class use another class as a field — choose based on whether the contained object can exist on its own
// B3.2.5 · Design Patterns

Design Patterns

Design patterns are proven, reusable solutions to recurring software design problems. They are best practices from experienced OOP developers — not code, but templates for solving common problems.

Singleton
Ensures a class has only one instance and provides global access to it.

Use for: database connections, settings managers, logging systems.
Factory
Provides an interface for creating objects without specifying the exact class. A factory decides which class to instantiate.

Use for: UI elements, document exporters, payment gateways.
Observer
One object (subject) notifies many observers when its state changes — a subscription model.

Use for: event listeners, news feeds, stock price alerts.

Why learn patterns? Design patterns give you a shared vocabulary — when you say "use a singleton here", every developer on the team immediately understands the structure and intent.

// B3.2.5 · Singleton Pattern

Singleton — One Instance Only

Only one instance of the class ever exists. Every call to "get the instance" returns the same object.

class DatabaseConnection: _instance = None # class-level variable holds the single instance def __new__(cls): if cls._instance is None: cls._instance = super().__new__(cls) cls._instance.connected = False return cls._instance def connect(self, url): self.url = url self.connected = True print(f"Connected to {url}") # Test: both variables point to the SAME object db1 = DatabaseConnection() db2 = DatabaseConnection() print(db1 is db2) # → True db1.connect("mysql://localhost/school") print(db2.url) # → mysql://localhost/school
①
__new__ controls object creation — if an instance exists, return it; otherwise create one
②
Guarantees that all parts of a program share one database connection — no wasted resources
// B3.2.5 · Factory Pattern

Factory — Object Creation Interface

A factory function/class creates objects without exposing the instantiation logic. The caller says what it wants; the factory decides how to create it.

class PrimaryButton: def render(self): return "[PRIMARY BUTTON]" class DangerButton: def render(self): return "[DANGER BUTTON]" class GhostButton: def render(self): return "[GHOST BUTTON]" # Factory function — caller doesn't need to know class names def create_button(style): buttons = { "primary": PrimaryButton, "danger": DangerButton, "ghost": GhostButton } return buttons[style]() btn = create_button("danger") print(btn.render()) # → [DANGER BUTTON]

Use case: Adding a new button type only requires adding a new class and updating the factory dictionary — the rest of the codebase is unaffected. This makes systems easy to extend.

// B3.2.5 · Observer Pattern

Observer — Subscribe & Notify

One Subject maintains a list of Observers. When the subject's state changes, it notifies all subscribed observers automatically.

class StockTicker: # Subject def __init__(self): self._observers = [] self._price = 0 def subscribe(self, observer): self._observers.append(observer) def set_price(self, price): self._price = price self._notify_all() # automatically notify all observers def _notify_all(self): for obs in self._observers: obs.update(self._price) class PhoneAlert: # Observer def update(self, price): print(f"📱 Alert: stock is now ${price}") class EmailAlert: # Observer def update(self, price): print(f"📧 Email: stock is now ${price}") ticker = StockTicker() ticker.subscribe(PhoneAlert()) ticker.subscribe(EmailAlert()) ticker.set_price(142.50) # → 📱 Alert: stock is now $142.5 # → 📧 Email: stock is now $142.5
// Summary · OOP Relationships

Class Relationships — Compared

IS-A
Inheritance — Dog is-a Animal. Child class extends parent class. Use class Dog(Animal). Promotes code reuse; parent changes affect all children.
HAS-A
strong
Composition — Car has-a Engine (strong). The engine is created and destroyed with the car. The contained object cannot exist independently.
HAS-A
weak
Aggregation — University has-a Student (weak). The student continues to exist even if the university closes. The contained object can exist independently.
Singleton
One instance only, globally shared. Database connections, app settings.
Factory
Centralized object creation. Easy to add new types without changing callers.
Observer
One-to-many notification. Subject changes → all observers auto-notified.
// Exam Strategy · B3.2

Key Exam Points

①
Inheritance vs Composition: Use inheritance for is-a relationships. Use composition for has-a relationships. Don't inherit just to reuse code if there is no logical is-a relationship.
②
Overriding vs Overloading: Overriding = subclass redefines parent method (same signature, different class). Overloading = same class, same name, different parameters.
③
Abstract classes: Cannot be instantiated. Must override all abstract methods in subclasses or the subclass is also abstract. Enforces a "contract".
④
Access modifiers in inheritance: private — not inherited directly; protected — accessible in subclasses but not externally; public — accessible everywhere.
⑤
Design patterns: Name the pattern, state its purpose, and give a real-world context. Don't just describe the code — explain why this pattern is appropriate for the scenario.
