// B3.1.1 · Fundamentals

What is OOP?

OOP is a programming paradigm that organizes code around objects — custom data types that bundle together data (attributes) and the functions that operate on that data (methods).

The Blueprint Analogy
A class is like an architectural blueprint — it defines the structure and behaviour, but is not a real object itself.

An object is like an actual house built from that blueprint — a real instance with its own specific values.
Why OOP?
Before OOP, programs were just sequences of instructions. OOP allows you to model real-world entities — a Person, BankAccount, Car — each knowing its own data and how to act on it.

Definition: Object-oriented programming — a form of programming that involves creating code for classes of objects, allowing many such objects to be created from a single code base, achieving a more modular and extensible software development process.

// B3.1.1 · Core Concepts

The Five Pillars of OOP

IB requires you to know these five fundamental concepts. B3.1 covers the first two in depth; B3.2 (HL) covers the rest.

①
Class — a template/blueprint defining attributes and methods
e.g. BankAccount defines that every account has a name and balance, and can deposit(), withdraw()
②
Object — a specific instance of a class with its own data values
e.g. amy = BankAccount("Amy") creates an object with name="Amy" and balance=0
③
Encapsulation — bundling data and methods together; hiding internal state SL
Other code interacts only through well-defined public methods — it cannot directly modify internal variables
④
Inheritance — a child class inherits attributes and methods from a parent class HL — B3.2
e.g. SavingsAccount inherits from BankAccount and adds interest-calculation functionality
⑤
Polymorphism — the same method name behaves differently in different classes HL — B3.2
e.g. toString() on a BankAccount prints balance; on a Person prints name and age
// B3.1.1 · Evaluate

Advantages & Disadvantages of OOP

The IB command term Evaluate means you must discuss both sides.

ADVANTAGES
+
Modularity — code is organized around objects, easier to manage large projects
+
Reusability — a class can be written once, imported and reused across many programs
+
Encapsulation — internal data is protected; prevents unintended side effects
+
Scalability — larger teams can work in parallel on separate classes
+
Maintainability — changes to a class don't necessarily break other code
DISADVANTAGES
−
Learning Curve — concepts like classes, inheritance, polymorphism are complex for beginners
−
Complexity — simple problems can become unnecessarily complex when forced into OOP
−
Overuse — deep inheritance hierarchies can make code hard to understand and debug
−
Overhead — OOP typically introduces additional memory and processing overhead

Conclusion: OOP is excellent for large, complex systems where modularity and collaboration matter. For simple scripts or small programs, a procedural approach may be simpler and more efficient.

// B3.1.2 · UML Class Diagrams

Designing with UML

A UML (Unified Modelling Language) Class Diagram is the standard way to plan a class before writing code. It has three sections:

GENERIC TEMPLATE
ClassName
− attribute: type
− attribute: type
+ ClassName(params)
+ method(params): returnType
+ method(params): returnType
Row 1
Class name (capitalized noun)
Row 2
Instance variables with data types
Row 3
Methods including constructor, params, and return type
EXAMPLE: BankAccount
BankAccount
− name: string
− balance: float
+ BankAccount(name: string)
+ deposit(amount: float): void
+ withdraw(amount: float): void
+ transfer(amount: float, recipient: BankAccount): void
+ toString(): string

Visibility: − means private (only accessible inside the class). + means public (accessible from outside).

// B3.1.4 · Constructors & Classes

Writing a Class & Constructor

The constructor is a special method that runs automatically when an object is created. It initializes the object's instance variables.

PYTHON
class BankAccount: def __init__(self, name): # Instance variables self.name = name self.balance = 0.0 def deposit(self, amount): self.balance += amount def withdraw(self, amount): self.balance -= amount def __str__(self): return f"Account {self.name}: ${self.balance}" # Create objects (instantiation) amy = BankAccount("Amy") amy.deposit(100) print(amy) # Account Amy: $100.0
JAVA
public class BankAccount { // Instance variables private String name; private double balance; // Constructor public BankAccount(String name) { this.name = name; this.balance = 0.0; } public void deposit(double amount) { balance += amount; } public String toString() { return "Account " + name + ": $" + balance; } } // In Main: BankAccount amy = new BankAccount("Amy"); amy.deposit(100);

Python: Use self to refer to the current object. The constructor is always named __init__. Java: Use this to distinguish instance variables from parameters with the same name.

// B3.1.4 · Instantiation

Creating Objects & Arrays of Objects

Instantiation is the process of creating a specific instance of a class (an object). Memory is allocated and the constructor runs.

PYTHON — SINGLE & ARRAY
# Single object acc = BankAccount("Neo") print(acc) # List of objects — Method 1 accounts = [ BankAccount("Amy"), BankAccount("Brian"), BankAccount("Clare") ] # List of objects — Method 2 accounts = [] for i in range(3): name = input("Enter name: ") accounts.append(BankAccount(name)) # Traverse and call methods for acc in accounts: print(acc)
JAVA — SINGLE & ARRAY
// Single object BankAccount acc = new BankAccount("Neo"); System.out.println(acc); // Array of objects BankAccount[] accounts = new BankAccount[3]; accounts[0] = new BankAccount("Amy"); accounts[1] = new BankAccount("Brian"); accounts[2] = new BankAccount("Clare"); // Traverse for (BankAccount a : accounts) { System.out.println(a); }

Naming Convention: Class names use PascalCase (BankAccount). Method names and variable names use camelCase in Java (depositAmount) and snake_case in Python (deposit_amount). Array/list names should be plural of the class name.

// B3.1.5 · Encapsulation

Encapsulation & Information Hiding

Encapsulation bundles data (attributes) and the methods that operate on them into a single unit (the class), while information hiding restricts direct access to an object's internal data.

❌ Without Encapsulation
If balance is public, any code can do:
amy.balance = -999999

This breaks the integrity of the object's data — a negative balance could be set directly without any validation.
# BAD — direct attribute access amy = BankAccount("Amy") amy.balance = -99999 # No guard!
✓ With Encapsulation
balance is private. Outside code must use withdraw(), which can validate the amount and prevent invalid states.

Internal data is protected. The class controls how its data can be modified.
def withdraw(self, amount): if amount > self.balance: print("Insufficient funds!") else: self.balance -= amount

Principle: Programmers only interact with a class through its public interface (the public methods). They do not need to know — or have access to — the internal data. This prevents accidental corruption and simplifies maintenance.

// B3.1.5 · Access Modifiers

Public, Private & Getters/Setters

public
Accessible from anywhere — inside the class, from other classes, from main program.
Use for methods that form the public interface of the class (e.g. deposit(), toString()).
private
Accessible only from within the class itself.
Use for all instance variables and helper methods that should not be touched directly (e.g. balance, name).
GETTER & SETTER METHODS — Controlled Access to Private Data
# Python getter/setter class BankAccount: def __init__(self, name): self.__name = name # private self.__balance = 0 # private def get_balance(self): # getter return self.__balance def get_name(self): # getter return self.__name
// Java getter public class BankAccount { private String name; private double balance; public String getName() { return this.name; } public double getBalance() { return this.balance; } }

Python Convention: Use double underscore prefix (__balance) to make attributes private (name-mangling). In Java, use the private keyword explicitly.

// B3.1.3 · Static vs Non-Static

Static vs Non-Static (Instance)

The key distinction: does this belong to the class as a whole, or to each individual object?

Non-Static (Instance)
Belongs to each individual object. Each object has its own copy.

e.g. balance and name in BankAccount — Amy's balance is different from Brian's.
Static (Class)
Belongs to the class itself — shared by all objects. Only one copy exists.

e.g. interestRate that applies to all bank accounts equally; accountCount tracking how many accounts exist.
# Python static (class variable) class BankAccount: interest_rate = 0.03 # class var account_count = 0 # class var def __init__(self, name): self.name = name # instance self.balance = 0 # instance BankAccount.account_count += 1 # Access class variable directly print(BankAccount.interest_rate)
// Java static public class BankAccount { private static double interestRate = 0.03; private static int accountCount = 0; private String name; // instance private double balance; // instance public static int getCount() { return accountCount; } } // Call without an object: BankAccount.getCount();

Rule of thumb: If every object needs its own separate copy → instance variable. If all objects share the same value, or it doesn't relate to any specific object → static/class variable.

// Worked Example

Complete Example: Student Class

Applying all B3.1 concepts together:

UML DESIGN
Student
− name: string
− grade: float
+ studentCount: int (static)
+ Student(name: string)
+ setGrade(g: float): void
+ getGrade(): float
+ getName(): string
+ toString(): string
class Student: student_count = 0 # static def __init__(self, name): self.__name = name # private self.__grade = 0.0 # private Student.student_count += 1 def set_grade(self, g): if 0 <= g <= 100: self.__grade = g def get_grade(self): return self.__grade def get_name(self): return self.__name def __str__(self): return f"{self.__name}: {self.__grade}" s1 = Student("EJ") s1.set_grade(92) print(s1) # EJ: 92 print(Student.student_count) # 1

Notice: __grade is private — validated in set_grade() to prevent invalid values. student_count is a class variable — shared and incremented each time any Student is created.

// Review · B3.1

Review & Key Takeaways

Classes & Objects
A class is a blueprint. An object is an instance. A class defines attributes (data) and methods (behaviour). Multiple objects can be created from one class, each with their own data.
UML Class Diagrams
Three-row table: class name → attributes with types → methods with params/return. − = private, + = public. Design before coding.
Constructors
Automatically called at instantiation. Initializes instance variables. Python: __init__(self, ...). Java: same name as class. Use self / this to refer to the current object.
Encapsulation
Make attributes private. Provide public getter/setter methods for controlled access. Prevents invalid data and unintended side effects. Core principle of safe OOP design.
Static vs Instance
Static = belongs to the class (shared). Instance = belongs to each object (unique copy). Use static for shared constants or counters; instance for per-object data.
Evaluate OOP
Pros: modularity, reusability, encapsulation, scalability. Cons: learning curve, complexity, overhead. Best for large, complex, collaborative projects.
