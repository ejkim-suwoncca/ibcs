// Criterion A · 4 Marks · 300 words

Problem Specification

Describe the problem, define measurable success criteria, and justify your computational context. This criterion shapes every criterion that follows.

1–2
Outlines a problem scenario. States limited success criteria. Outlines the computational context.
3–4
Describes the problem in terms of measurable solution requirements. States appropriate success criteria. Explains and justifies the choice of computational context.
Good Success Criteria
"The solution will be able to input, edit and delete customer records from the database."

"The solution will display appropriate warning messages for invalid input."

Use SMART: Specific, Measurable, Achievable, Relevant, Time-bound.
Inadequate Success Criteria
"The code will compile."
"The code will run without crashing."
"The code will use loops."
"The program will be aesthetically pleasing."

These are not testable against the core functionality.

Key point: Your success criteria are referenced in Criterion B (planning), D (testing), and E (evaluation). Weak criteria here cascade into problems throughout the entire IA.

// Criterion B · 4 Marks · 150 words

Planning & Decomposition

Break the problem into manageable parts and create a realistic timeline for planning, designing, developing, testing, and evaluating.

1–2
Partial decomposition of the problem. Plan addresses some of the success criteria.
3–4
Reasonable decomposition of the problem. Plan addresses all the success criteria.
Decomposition tools
Structure chart, use-case diagram, data flow diagram, or bulleted list with indentation. Identify each sub-problem and connect it to a success criterion.
Planning tools
Gantt chart or Agile chart. Must show all 5 stages: Planning → Designing → Developing → Testing → Evaluating — with realistic timeframes.

Common mistake: The plan is NOT a process journal written after development. The IB requires the plan to be written before the computational solution is completed — it should guide development, not describe it retrospectively.

// Criterion C · 6 Marks · 150 words

System Overview

Construct a system model clear enough that a third party could recreate the product, plus algorithms and a testing strategy.

MarksDescription
1–2Outlines a limited system model. Identifies algorithms. Identifies a testing strategy for at least one success criterion.
3–4Constructs an incomplete system model. Constructs algorithms leading to partial functionality. Outlines testing for at least three success criteria.
5–6Constructs a complete system model. Constructs algorithms enabling the product to perform. Describes a testing strategy aligned with all success criteria.
Recommended Diagrams
UI wireframes (mandatory) · ERD (if database) · Data dictionary · UML class diagram (if OOP) · Flowcharts (if procedural) · Component diagram (if hardware)
Algorithms
Present in natural language, flowcharts, or pseudocode. Only need to cover key components — not every function. Algorithms must be detailed enough to enable the product's functionality.
// Criterion C · Testing Strategy

Testing Strategy

Your documentation should contain two test tables — one functional, one structural.

Functional Testing
Tests that the application meets user requirements. Evaluates against all success criteria by providing inputs and examining outputs. Tests from the user's point of view.
Structural Testing
Tests the internal workings of the code. Verifies all conditional branches execute correctly and error-handling triggers appropriately. Uses valid, extreme, and invalid data.
Input Validation to Test
Presence check: Has a value been input?   Length check: Is the input an acceptable length?   Type check: Is the input the correct data type?   Format check: Does it follow complex formatting rules (dates, currency)?

Common mistake: Tests like "starts up within 10 seconds" or "does not crash" are not functional tests. Test actual requirements, e.g. "successfully adds record to the XXX table when a message is received."

// Criterion D · 12 Marks · 1000 words

Development · Implementation

The largest criterion — 40% of total marks. Document your code choices, algorithm implementation, and testing effectiveness.

MarksDescription
1–3Very limited functionality. No appropriate techniques. States choices and testing strategy.
4–6Limited functionality. At least one appropriate technique. Outlines choices. States testing effectiveness.
7–9Partial functionality. Some appropriate techniques. Explains choices. Describes testing effectiveness.
10–12Fully functional product. Appropriate techniques used. Evaluates algorithm choices. Justifies testing effectiveness.

Top-band shift: From 7–9 to 10–12, the shift is from explain/describe to evaluate/justify. Why were the techniques appropriate? Why was your testing approach optimal for this context? Show at least 3 test outcomes from both functional and structural testing.

// Criterion D · Techniques & Structure

Writing Criterion D

Structure Criterion D with subheadings based on success criteria — not as a narrative of development history.

Code
Include key code excerpts with comments. All code in documentation must be referenced to the full source code in the appendices. Present only code that is critical to the algorithm's functionality.
Tech
Justify technique choices. Techniques may include: loops, data structures, existing libraries, recursion, OOP, multi-threading, encryption, database integration. Explain why each is appropriate for your specific problem.
Test
Show testing results. The video demonstrates functionality and examples of tests in action. The documentation describes testing strategy and justifies its effectiveness — with at least 3 concrete test outcomes covering both functional and structural testing.

Common mistake: Blocks of code hundreds of lines long suggest inadequate decomposition. Variable names like person1, person2, person3 may indicate an array should have been used — and examiners notice this.

// Criterion E · 4 Marks · 400 words

Evaluation · Reflection

Evaluate how well each success criterion was met, and justify specific, actionable improvements.

1–2
States the extent to which success criteria were met. Describes improvements.
3–4
Evaluates (weighs strengths and limitations) the extent to which each criterion was met. Justifies improvements with reference to specific issues.
Good Improvements
Specific, justified improvements that address concrete issues. E.g. "Adding an array to persist characteristics when navigating between pages, addressing the data-loss issue observed in testing."
Weak Improvements
"Add more colour / more data / more functionality." Fixing criteria that weren't met (should have been developed). Adding a GUI to a CLI solution. These are vague and not justified.

Top tip: A table listing each success criterion with an evaluation of whether it was met, the evidence (from your tests), and any limitations — is an effective format for a 3–4 mark response.

// Pitfalls

Common Mistakes

A
Success criteria don't reference core functions. If you design a task manager without mentioning the core task-management features in the criteria, testing becomes impossible later. Every key function must appear somewhere in the criteria.
B
Plan written after the fact. The planning stage should precede development. Gantt or Agile charts backdated after coding is completed contradict the purpose of planning.
C
Technical errors in diagrams. Flowcharts with two arrows leaving a non-decision element; UML diagrams with incorrect relationship notation. Every diagram must comply with its standard.
D
Describing history instead of justifying choices. "I first tried X then changed to Y because Z" is narrative, not justification. Write about why your final approach is appropriate, with evidence from testing.
E
Overly simplistic evaluation. "Haven't run into any bugs or errors" is a statement, not an evaluation. Evaluate means "weigh strengths and limitations" — even a working feature can be evaluated for efficiency or robustness.
// Project Selection

Choosing Your Project

The IA does not need to relate to a specific syllabus topic. Choose something you genuinely care about and have the skills to build.

Good project qualities
Original idea with an element of innovation. Clearly defined problem that benefits from a computational solution. Feasible within your skill level and available time. Testable success criteria derivable from the problem.
Computational contexts
Language (Python, Java, etc.) + why it fits the project. Software framework chosen (JavaFX, Flask, etc.). Hardware requirements if any. Database engine if used. Network conditions if applicable.
Ask
Are there existing solutions, and why are they not effective? What is innovative or original about your approach? What are your resource and time constraints?

Avoid: Simple clones of existing apps without original ideas. Projects where all success criteria are trivially met from the start. Projects that are fundamentally beyond your available technical skills — quality of justification and testing matters more than complexity.
