// A1.3.1 · Role of the OS

What Does an OS Do?

The OS is the software layer between the user and hardware. It manages resources and provides services for all programs.

CPU Management
Allocates CPU time to processes efficiently.
Memory Management
Allocates/de-allocates RAM; manages virtual memory.
Storage Management
Organizes data on disks; handles file systems.
Device Management
Controls hardware via device drivers.

Well-known operating systems: Windows, macOS, Linux (desktop) · Android, iOS (mobile). Each abstracts hardware so apps don't need to know the hardware details.

// A1.3.2 · OS Functions — Overview

7 Key Functions

Memory Management
Loads apps into RAM. Uses virtual memory (disk space acting as RAM) when RAM runs low.
File Management
Hierarchical directory trees. Create, read, write, delete, rename files. Defragmentation keeps disks fast.
Device Management
Device drivers bridge OS ↔ hardware. Uses buffering, caching, spooling. Plug and Play auto-installs.
Scheduling
Decides which process runs on CPU and when. Enables multitasking. (Detail on next slides.)
Security
Authentication (login, biometrics), encryption, audit logs, malware protection (antivirus, firewall, IDS).
Accounting
Tracks CPU, memory, disk, network usage per process/user. Used for billing, quotas, performance monitoring.
GUI + Networking + Virtualization
Visual interface (windows, icons, menus). Manages network connections (TCP/IP). Runs multiple VMs via hypervisor.
// A1.3.2 · Memory Management

Memory Management in Depth

01
App launches → OS locates the .exe / .app file on disk
02
OS allocates RAM for code, data, and resources
03
OS isolates each process in its own memory space — no interference
04
If RAM fills up: OS uses virtual memory — spills low-priority processes to disk
Virtual Memory
Part of the HDD/SSD is used as overflow RAM. Much slower than real RAM, but keeps the system running. High-priority processes stay in RAM; low-priority move to disk.

Analogy: RAM is your desk — fast, limited space. Virtual memory is a filing cabinet — slower, but you can store more.

// A1.3.3 · Scheduling — Part 1

FCFS & Round Robin

First Come First Served (FCFS)
Processes execute in arrival order. Simple, non-pre-emptive — a process runs until complete before the next starts.
✓ Advantages ✗ Disadvantages
Simple to implement
Fair arrival order		 Convoy effect — short jobs stuck behind long ones
Higher average wait times
Round Robin (RR)
Each process gets a fixed time quantum. After it expires, the process goes to the back of the queue. Pre-emptive — no process monopolizes the CPU.
✓ Advantages ✗ Disadvantages
Fair CPU sharing
High responsiveness		 Quantum choice is critical
High context-switching overhead if quantum too short

Key term — Context switch: When the CPU saves a process's state and loads another. Necessary for multitasking but adds overhead.

// A1.3.3 · Scheduling — Part 2

Priority & Multilevel Queue

Priority Scheduling
Each process has a priority. Highest priority runs first. Priority can be static or dynamic. Ageing gradually increases waiting process priority to prevent starvation.
✓ Advantages ✗ Disadvantages
Critical tasks run immediately
Flexible, dynamic priority		 Starvation of low-priority processes
Priority inversion possible
Multilevel Queue Scheduling
Ready queue split into multiple queues (e.g. interactive, batch, background). Each queue has its own algorithm and priority level.
✓ Advantages ✗ Disadvantages
Tailored algorithms per queue
Responsive for interactive tasks		 Low queues can starve
Processes locked into queues
Q1 HIGH → Interactive → Round Robin
Q2 MED  → Batch     → FCFS
Q3 LOW  → Background → FCFS
// A1.3.4 · Interrupt Handling vs Polling

Interrupts vs Polling

Two ways for the CPU to find out when a device needs attention.

CriteriaInterrupt HandlingPolling
Mechanism Device signals CPU when it needs service → CPU pauses, runs ISR, resumes CPU repeatedly checks each device at set intervals
Best for Unpredictable, infrequent events (keyboard, network packet) Regular, predictable events (climate sensor, data logger)
CPU overhead Low when events are rare; high if interrupts are very frequent High — CPU checks constantly even when nothing happens
Power use More efficient — CPU idles between events Less efficient — CPU always active
Latency Immediate response; variable if many interrupts Controlled, predictable response time

Analogy: Interrupts = doorbell (acts when something happens). Polling = checking the door every 5 minutes whether someone is there or not.

// A1.3.3 · Quick Reference

Scheduling — At a Glance

AlgorithmHow it worksPre-emptive?Main risk
FCFS Run in arrival order, each to completion No Convoy effect — long jobs block short ones
Round Robin Fixed time quantum, then back of queue Yes Too-short quantum → high context-switch overhead
Priority Highest priority runs first; ageing prevents starvation Yes Starvation of low-priority; priority inversion
Multilevel Queue Separate queues per process type, each with own algorithm Yes Lower queues starve; processes locked into queues

Common mistake: Context switching enables multitasking but adds overhead — too much switching reduces performance.

// A1.3.4 · Real-World Examples

When to Use Each?

USE INTERRUPTS WHEN…
→
Keyboard / Mouse: each keystroke fires an interrupt for instant response
→
Network packet arrives: NIC interrupts CPU to process data immediately
→
Disk I/O completes: disk signals CPU so it can multitask during the wait
→
Airbag sensor: collision detection needs an immediate, critical response
USE POLLING WHEN…
→
Temperature sensor: checked at regular intervals in climate control
→
Data logger: writes to disk at fixed intervals — polling is simpler
→
Industrial robot: performs checks at precise, predictable intervals
→
Simple kiosk: low activity — polling is sufficient and easier to implement

Key point: Neither is universally better. Context determines the right choice — event frequency, power source, and response time requirements all matter.
