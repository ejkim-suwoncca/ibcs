// Motivation · The Problem Pipelining Solves

The Problem: Idle Stages

Without pipelining, the CPU must finish one instruction completely before starting the next. This leaves hardware stages sitting idle and wastes potential throughput.

Non-Pipelined (Sequential)
Instruction A: Fetch → Decode → Execute → Write Back
Only then does Instruction B begin.

While instruction A is being decoded, the Fetch hardware sits idle. Wasteful. Like a carwash where only one car can be in any stage at any time.
Pipelined (Overlapping)
As soon as Instruction A finishes the Fetch stage, Instruction B enters the Fetch stage immediately — while A moves to Decode.

Multiple instructions are in different stages simultaneously. No stage ever sits idle. Much higher throughput.

Carwash analogy (from the textbook): Stages are initial wash, detailed clean, rinse, dry — each 5 min. Non-pipelined: 2 cars = 8 × 5 = 40 min (20 min/car). Pipelined: 2 cars = 5 × 5 = 25 min (12.5 min/car). The efficiency gain scales with more instructions.

// A1.1.6 · CPU Pipeline Stages

The Five Pipeline Stages

The CPU's instruction cycle is broken into five distinct stages, each handled by dedicated hardware.

F
Fetch — The instruction is retrieved from memory. The Program Counter (PC) holds the address; the instruction is placed in the Instruction Register (IR) and the PC is incremented.
D
Decode — The instruction is interpreted by the Control Unit. It determines what operation is required (ADD, MOV, JMP…) and which registers or memory locations are involved.
E
Execute — The ALU or other hardware unit carries out the operation — arithmetic, logical comparison, branching, etc. This is where the actual computation happens.
M
Memory Access — Any required data is read from or written to memory. Only needed for load/store instructions; other instructions may skip or do nothing at this stage.
W
Write Back — The result of the operation is written back to the appropriate CPU register. This completes the instruction's processing.

// A1.1.6 · Multi-Core Architectures

Multi-Core Processors

A multi-core processor has multiple independent CPU cores on a single chip. Each core has its own pipeline and can execute instructions completely independently.

Independent Execution
Each core maintains its own set of pipelines. Core 1 can be executing a word-processing task while Core 2 handles background virus scanning — simultaneously, with no interference.

This is the main advantage over single-core CPUs: true parallelism for independent tasks.
Parallel Execution
Cores can also work together on a single large task that has been divided into sub-tasks. E.g. rendering different parts of a 3D scene, or processing different sections of a large dataset simultaneously.

This requires software designed for parallel execution (multi-threaded programming).

Analogy: Single-core with pipelining = one carwash team working efficiently on multiple cars in sequence. Multi-core = multiple independent carwash teams, each with their own efficient pipeline, all working simultaneously.

// Combined Effect

Pipelining + Multi-Core Together

Modern processors combine both techniques: each core has its own pipeline, and multiple cores run in parallel. This combination significantly boosts overall computational efficiency.

Core 1 — Pipeline
Cycle 1: F-A
Cycle 2: D-A F-B
Cycle 3: E-A D-B F-C
Cycle 4: M-A E-B D-C …
Core 2 — Pipeline
Cycle 1: F-X
Cycle 2: D-X F-Y
Cycle 3: E-X D-Y F-Z
Cycle 4: M-X E-Y D-Z …
Result
Both cores run completely independently. By cycle 4, 8 different instructions are in various stages of execution simultaneously — 4 per core. Throughput scales with core count.

Real-world impact: Handling complex, resource-intensive tasks (video editing, machine learning, gaming, simulations) becomes practical because the workload can be split across cores, and each core processes its share via its own pipeline.

// Pipeline Challenges · Hazards

Pipeline Hazards

Pipelining isn't perfect. Certain situations cause the pipeline to stall (insert empty cycles, called "bubbles"), reducing efficiency.

Data Hazard
Instruction B needs the result of instruction A, but A hasn't finished yet. The pipeline must wait for A to write back before B can execute. Solution: forwarding (bypassing) or stalling.
Control Hazard
A branch instruction (if/jump) means the CPU doesn't know which instruction to fetch next until the branch resolves. It may have already loaded wrong instructions into the pipeline. Solution: branch prediction.
Structural Hazard
Two instructions need the same hardware resource at the same time (e.g. both need memory access). One must wait. Solution: duplicating resources (separate instruction and data caches).

IB note: You don't need to know all hazard types in depth — but you should be aware that pipelining can encounter hazards that temporarily reduce efficiency. Being able to explain that pipelines don't always run at full speed due to dependencies is enough.

// Evaluation

Benefits & Limitations of Pipelining

BENEFITS
✓
Higher throughput: Close to one instruction completed per clock cycle when the pipeline is full.
✓
Better hardware utilization: All pipeline stages are kept busy; no hardware sits idle between instructions.
✓
No clock-speed increase needed: Performance improves through parallelism rather than requiring faster (and more expensive) hardware.
✓
Scales with multi-core: Each additional core adds another full pipeline, multiplying total throughput.
LIMITATIONS
✗
Pipeline hazards: Data dependencies, branch instructions, and resource conflicts can cause stalls, reducing actual throughput below the theoretical maximum.
✗
Increased hardware complexity: Pipeline registers, forwarding units, and hazard detection circuits add design complexity and cost.
✗
Branch misprediction cost: When the CPU predicts a branch incorrectly, it must flush incorrect instructions from the pipeline — a significant performance penalty.
✗
Limited by slowest stage: The clock cycle is constrained by the slowest pipeline stage. No stage can be skipped even if it has nothing to do.
// Real World

Pipelining in Practice

Modern CPU Cores
Modern processors (Intel Core, AMD Ryzen) use deep pipelines with many stages. High-end CPUs have 14–20+ pipeline stages. Combined with 4–16+ cores, they can process enormous numbers of instructions simultaneously.
GPU Pipelines
GPUs contain thousands of simpler cores, each with its own pipeline — designed for parallel workloads like rendering pixels or training neural networks, where the same operation is applied to massive datasets.
Machine Learning
Training ML models involves billions of matrix multiplications — highly parallelizable. GPUs with many pipelined cores dramatically reduce training time compared to CPUs. (Connects to Theme A4.)
Everyday Devices
Smartphones, tablets, and laptops all use multi-core, pipelined CPUs. A modern smartphone CPU may have 4–8 cores, each pipelined, running thousands of background and foreground tasks simultaneously.

Connecting to A4: The reason GPUs are essential for machine learning is that they have thousands of smaller pipelined cores — each efficiently processing data — running in parallel. Pipelining and multi-core architecture are the foundation that makes AI computationally feasible.

// Review · A1.1.6

Summary & Key Takeaways

What is Pipelining?
A technique that overlaps the execution of multiple instructions by having each pipeline stage work on a different instruction simultaneously. Like a carwash with multiple cars in different stages at once.
The 5 Stages
Fetch → Decode → Execute → Memory Access → Write Back. Each stage is handled by dedicated hardware. When the pipeline is full, a new instruction completes every clock cycle.
Write-Back Stage
The final stage: the result of the instruction is written back to a CPU register. This is what "completes" the instruction. Only after write-back is the result available for use by subsequent instructions.
Multi-Core + Pipelining
Each core in a multi-core processor has its own independent pipeline. Cores can work independently (different tasks) or in parallel (cooperating on one task). Combined effect: massive throughput improvement.
Hazards
Data hazards, control hazards, and structural hazards can cause pipeline stalls. Modern CPUs use forwarding, branch prediction, and separate caches to minimize stalls.
Evaluate
Benefits: higher throughput, better utilization, no clock-speed increase needed. Limitations: hazards cause stalls, increased hardware complexity, limited by slowest stage.

One-sentence summary: Pipelining improves CPU performance by keeping all hardware stages busy simultaneously — processing different instructions at different stages in the same clock cycle — and multi-core architectures multiply this effect by running independent pipelines in parallel.
