// The Big Picture

Why Do We Need Translation?

Computers only understand machine code (0s and 1s). Humans write in high-level languages. Translation bridges the gap.

// Interpreters

How an Interpreter Works

Mechanics
Reads source code line by line, translates each line to machine code, and executes it immediately. No separate executable file is saved. The translation happens fresh every time the program runs.
Use Cases
• Rapid development & debugging (see results immediately)
• Scripting, web development
• Education — beginners see output in real time
• Cross-platform (same code runs anywhere an interpreter is installed)
• Languages: Python, JavaScript, Ruby
✓ Advantages ✗ Disadvantages

Easy to test and debug interactively

Highly portable across platforms

Errors shown as they occur — helpful for learning		 Slower — retranslates every run

Errors only found when that line is reached at runtime

Analogy: A live translator who reads each sentence of a book aloud — useful immediately but must retranslate every time.

// Compilers

How a Compiler Works

Mechanics
Reads the entire source code in one pass through stages: lexical analysis → syntax analysis → semantic analysis → optimization → code generation. Produces a standalone executable in machine code. Once compiled, no further translation is needed.
Use Cases
• Performance-critical applications
• System software, operating systems
• Video games, real-time systems
• Security-sensitive code (harder to reverse-engineer)
• Languages: C, C++, Java (→ bytecode)
✓ Advantages ✗ Disadvantages
Fast execution — translated once, runs instantly

All syntax errors caught before the program runs

More secure — source code not needed at runtime

Can be optimized for target hardware		 Slow to compile — large programs take time

All errors must be fixed before you can run anything

Less portable — may need recompiling per platform

Analogy: A translator who first reads the whole book and produces a finished translation — you wait longer upfront, but reading is seamless after.

// Advanced Approaches

Bytecode Interpreters & JIT

Bytecode Interpreter
Source code is first compiled into bytecode — a compact, platform-independent intermediate form. The bytecode then runs on a Virtual Machine (VM) rather than directly on hardware.

The same bytecode runs on any platform with the right VM.

Examples: Java → JVM, Python → .pyc files

Just-In-Time (JIT) Compilation
Starts by interpreting bytecode. As the program runs, the JIT compiler identifies "hot spots" — frequently executed sections — and compiles them to machine code on the fly. That compiled version is cached for future use.

Result: portability of bytecode plus near-compiled speed.

Examples: Java JVM, .NET runtime, JavaScript in browsers

Key idea: JIT gets faster the longer a program runs — hot spots keep getting optimized. Ideal for servers and long-running apps.

// Evaluation · Error Detection

When Are Errors Caught?

One of the biggest practical differences between translation methods.

Compiler — Before execution
Analyses the entire code upfront. All syntax and many semantic errors are caught and must be fixed before the program can run at all. Fewer surprises at runtime → more stable final product.
Interpreter — At runtime
Errors are discovered only when the problematic line is executed. Useful for quick iteration during development, but a bug in a rarely-run code path may go unnoticed until it crashes in production.
Bytecode Interpreter — Mixed
Some errors caught at the bytecode compilation stage, but others still surface at runtime. A middle ground — fewer runtime errors than a pure interpreter.
JIT — Runtime + dynamic fix
Errors can still occur at runtime, but the JIT compiler's repeated optimization of hot spots can detect and improve issues in frequently executed paths over time.

Common mistake: Don't confuse "compilers catch errors" with "compilers catch ALL errors." They catch syntax and some semantic errors — logic errors still slip through.

// Evaluation · Scenario Examples

Which to Choose?

The right translation method depends on the project's needs.

Rapid Development & Testing
→ Use an Interpreter

Example: A startup prototyping a web app in Python. The team needs to test and iterate fast — no compilation delay, errors surface immediately.

Languages like Python and JavaScript let developers see results right away.
Performance-Critical Applications
→ Use a Compiler

Example: A real-time trading system in C++ requiring minimal latency. The entire codebase is compiled to optimized machine code for maximum speed.

Also: game engines, OS kernels, embedded systems.
Cross-Platform Development
→ Use Bytecode / JIT

Example: An enterprise Java app that must run on Windows, macOS and Linux. Java compiles to bytecode → JVM handles each platform. JIT then optimizes hot paths per platform.

Also: Android apps, web services, .NET apps.

Tip: In the exam, always justify your choice using at least two criteria — e.g. "An interpreter is suitable because it allows rapid testing and the code needs to run on multiple platforms."

// Summary

What We Covered

Interpreter
Line-by-line translation + execution. Immediate start, slower overall. Errors at runtime. High portability. Best for: scripting, rapid dev, education. (Python, JS)
Compiler
Full translation upfront → standalone executable. Slow to compile, fast to run. All errors before execution. Low portability. Best for: performance-critical, system software. (C, C++)
Bytecode Interpreter
Source → bytecode → runs on VM. Platform-independent. Moderate speed. Mixed error detection. Best for: cross-platform enterprise/mobile apps. (Java JVM)
JIT Compilation
Bytecode + hot-spot compilation at runtime. Portable + fast (gets faster over time). Best for: long-running servers, dynamic web apps. (JVM, .NET, V8)

Big picture: All four methods translate the same thing — high-level code into machine-executable instructions. They differ in when translation happens and the trade-offs that follow.
