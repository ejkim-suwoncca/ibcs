// A1.3.5 · Multitasking

What is Multitasking?

Multitasking allows a system to perform multiple tasks seemingly simultaneously by rapidly switching between them — each process receives CPU time in turn.

The OS must manage
CPU time — which process runs and for how long
Memory — each process needs its own space
I/O devices — shared peripherals must be co-ordinated
Context switching — saving and restoring process state
Three HL Challenges
Task Scheduling — deciding execution order and duration
Resource Contention — multiple processes competing for the same resource
Deadlock — processes permanently blocked waiting for each other

Key idea: The CPU does not truly run multiple processes at the same instant — it switches so fast that it appears simultaneous. The OS scheduler controls this process.

// A1.3.5 · Task Scheduling

Task Scheduling & Context Switching

The OS scheduler decides which process gets CPU time and for how long, aiming to maximise utilisation and responsiveness.

Scheduling Algorithms (recap)
FCFS: simple but can cause convoy effect
Round Robin: fair time-slice rotation
Priority: urgent tasks first; risk of starvation
Multilevel Queue: separate queues per process type
Context Switching
When the CPU switches from one process to another it must:
1. Save the current process state (registers, PC, memory pointers)
2. Load the saved state of the next process

Each switch adds overhead — too many switches slow the system down.

Common mistake: Don't say context switching "enables multitasking" without noting the overhead it introduces. Excessive context switching reduces overall performance.

// A1.3.5 · Resource Contention

Resource Contention & Mutual Exclusion

Resource contention occurs when multiple processes compete for the same resource (CPU, memory, I/O device) at the same time.

Mutual Exclusion
Only one process may access a shared resource (called the critical section) at a time.

Techniques: Semaphores, locks, monitors.

Prevents data corruption when two processes try to write simultaneously.
Problems from Poor Management
Starvation: a process is repeatedly denied resource access — it never gets CPU time

Priority Inversion: a low-priority process holds a resource that a high-priority process needs, effectively blocking the high-priority task

Semaphore example: Two processes need to write to the same log file. A binary semaphore (0 or 1) acts as a lock — Process A sets it to 0 (locks), writes, then sets it back to 1 (releases). Process B waits until it is 1 before writing.

// A1.3.5 · Deadlock

Deadlock — Four Conditions

A deadlock occurs when a set of processes are permanently blocked — each is waiting for a resource held by another in the set. All four conditions must hold simultaneously.

1 · Mutual Exclusion
Resources cannot be shared — only one process can use a resource at a time.
2 · Hold and Wait
A process holds at least one resource while waiting to acquire additional resources held by others.
3 · No Pre-emption
Resources cannot be forcibly taken from a process — they must be released voluntarily.
4 · Circular Wait
Process A waits for B, B waits for C, C waits for A — a cycle of waiting with no exit.

Analogy: Four cars arrive at a four-way stop simultaneously. Each car is waiting for the car on its right to move. Nobody moves — deadlock.

// A1.3.5 · Deadlock Handling

Deadlock — Prevention, Avoidance & Recovery

PREVENT
Deadlock Prevention — design the system so at least one of the four conditions can never hold. Example: require processes to request all resources at once (eliminates "hold and wait").
AVOID
Deadlock Avoidance (Banker's Algorithm) — the OS only grants a resource if doing so keeps the system in a "safe state". If granting would risk deadlock, the process must wait.
DETECT
Deadlock Detection — allow deadlocks to occur but periodically check for circular waits using a resource allocation graph. If a cycle is found, take recovery action.
RECOVER
Deadlock Recovery — once detected, the OS can: terminate one or more deadlocked processes, or pre-empt resources from a process (roll back to a safe checkpoint) to break the cycle.

Key exam point: You must know the four necessary conditions for deadlock AND at least two strategies for handling it. Naming the strategy is not enough — explain the mechanism.

Common mistake: Don't just say "deadlock is when processes are stuck" — explain specifically that it involves circular resource dependency and why it cannot resolve itself.

// A1.3.6 · Control Systems

What is a Control System?

A control system is a set of components that work together to regulate a process or machine — taking inputs from the environment, processing them, and producing outputs that adjust behaviour.

Core Concept
Control systems are used to automate and regulate processes without continuous human intervention. They are fundamental in manufacturing, robotics, transport, home automation and environmental monitoring.
Key Components (A1.3.6)
Input — desired condition or target
Process — the controller deciding what to do
Output — the physical action taken
Feedback — monitoring the output and adjusting
Sensors, Actuators, Transducers

Why it matters for IB: A1.3.6 links directly to real-world applications (A1.3.7) and to ethical/societal questions about automation. Expect questions asking you to identify components from a scenario description.

// A1.3.6 · Core Model

Input → Process → Output → Feedback

Input
The desired target or reference condition. Could be set by a user (e.g. desired temperature = 22°C) or detected from the environment. Serves as the reference point the system aims to achieve.
Process (Controller)
The controller compares the desired input with the current state (from feedback) and computes the necessary action to minimise the difference (error) between them.
Output
The physical result produced — e.g. the heater switches on, the motor speed changes, a valve opens. The output is what we can observe and measure.
Feedback
Sensors measure the actual output and send this data back to the controller. The controller adjusts its actions based on this real-time information — keeping the system on target.

Key insight: Without feedback, the system is "blind" — it cannot correct errors. Feedback is what makes a control system adaptive.

// A1.3.6 · Components

Key Components of a Control System

Controller
The "brain" of the system. Processes input and feedback; computes the required output action using the control algorithm. Compares desired value (set point) with actual value (from sensors) and sends correction commands to actuators. Examples: thermostat circuit, microprocessor, PLC (Programmable Logic Controller).
Sensors
The "eyes and ears". Devices that detect and measure physical quantities — temperature, pressure, speed, light, motion — and convert them into electrical signals for the controller. Examples: thermistor, pressure sensor, IR sensor, camera, LiDAR.
Actuators
The "muscles". Convert the controller's commands into physical actions — mechanical motion, opening a valve, turning a motor, activating a heating element. Examples: electric motor, solenoid valve, servo, relay, speaker.
Transducers
Devices that convert one form of energy to another, bridging sensors/actuators and the controller. A microphone converts sound → electrical signal. A speaker converts electrical → sound. A pressure transducer converts mechanical pressure → voltage the controller can read.
Control Algorithm
The logic (set of mathematical rules) that determines the correct output from the input and feedback. Ranges from simple proportional control ("adjust in proportion to the error") to advanced PID control (Proportional-Integral-Derivative, which considers past, present and future error).

Common mistake: Don't oversimplify. Sensors and actuators have distinct roles. A sensor detects, a transducer converts, an actuator acts. In exams, identify which component does which job from the scenario.

// A1.3.7 · Example 1

Home Thermostat

Input
User sets desired temperature (e.g. 22°C)
Sensor
Thermistor continuously monitors room temperature
Controller
Compares actual vs desired; determines whether to heat or cool
Actuator
Switches boiler/AC unit on or off
Feedback
Temperature sensor feeds updated reading back — controller adjusts continuously
System Type
Closed-loop — feedback is essential. Without it, the boiler would keep running regardless of actual temperature.

Control algorithm: Simple ON/OFF (bang-bang) — if temp < set point, turn on heat. If temp ≥ set point, turn off.

Transducer note: The thermistor converts temperature (physical quantity) → electrical resistance → voltage signal that the controller can read. This is a transducer function.

// A1.3.7 · Example 2

Autonomous Vehicles

Self-driving cars integrate multiple sensor types, an AI-powered controller and multiple actuators — one of the most complex real-world control systems.

Sensors (Input)
Cameras, LiDAR, radar, GPS, ultrasonic sensors — detect obstacles, road markings, traffic signals, speed, position
Controller (Process)
AI algorithms process sensor data in real time to decide speed, steering direction, braking force
Actuators (Output)
Steering system, acceleration (throttle), braking — execute the controller's commands physically

Feedback: Sensors continuously re-scan the environment, feeding updated data to the controller — the vehicle adapts to sudden obstacles, lane changes, pedestrians in real time.

Transducers: LiDAR converts laser pulses → distance measurements; cameras convert light → digital images. Both are transducers bridging physical world and digital controller.

// A1.3.7 · Examples 3 & 4

Washing Machine & Traffic Signals

👕 Automatic Washing Machine
Sensors
Water level sensor, load-size sensor, cycle-progress sensor
Controller
Determines fill amount, wash duration, spin speed based on load
Actuators
Water valve, drum motor, drain pump, heating element
Feedback
Water level sensor stops fill when target reached; load sensor adjusts spin cycle intensity

System type: Closed-loop — feedback ensures the correct water level and spin speed for each load.

🚦 Traffic Signal Control
Sensors
Inductive loop sensors / cameras detect vehicle presence and count; pedestrian push buttons
Controller
Determines optimal timing and sequence — extends green for congested lanes, triggers pedestrian crossing
Actuators
Traffic lights (LED arrays) — red, amber, green for each direction
Feedback
Real-time vehicle counts feed back to controller — adapts signal timing dynamically to reduce congestion

System type: Closed-loop — without feedback, lights would use fixed timing regardless of actual traffic flow.

// Exam Guidance

Answering A1.3 HL Exam Questions

For Multitasking / Deadlock Questions
✓ Name the challenge (resource contention, deadlock, priority inversion)
✓ Define it precisely (e.g. deadlock = circular wait where each process holds a resource the next needs)
✓ Name the OS mechanism used to handle it (semaphore, prevention strategy, etc.)
✓ Give a brief concrete example
For Control System Questions
✓ Identify all components present in the scenario
✓ Correctly label sensor vs actuator vs transducer
✓ State whether open or closed loop and explain why
✓ Describe the feedback loop: what is measured? what is adjusted?
✓ Name what the controller decides

Common mistake: Confusing transducer with sensor. A transducer converts energy form; a sensor detects a physical quantity. Many sensors include transducer functionality — but the transducer specifically handles the energy conversion step.
