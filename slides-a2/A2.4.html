// A2.4.1 · Firewalls

How Firewalls Work

A firewall inspects and filters network traffic based on rules — allowing legitimate traffic and blocking threats.

How a Firewall Filters
Whitelist — only allow explicitly permitted traffic (deny everything else by default)

Blacklist — block known bad traffic, allow everything else

Rules — based on source IP, destination IP, port number, protocol (TCP/UDP)

Example rule: Block all incoming TCP connections on port 23 (Telnet — insecure protocol)
Types of Firewalls
Packet filtering — checks packet headers only; fast but limited

Stateful inspection — tracks connection state; more intelligent filtering

Application layer — inspects actual content of traffic (e.g., HTTP); most powerful

Hardware firewall — physical device (e.g., in a router)
Software firewall — program on a device (e.g., Windows Firewall)
Strengths of Firewalls
First line of defence against external attacks
Can block known malicious IPs and ports
Enforces network security policy consistently
NAT (built into many firewalls) hides internal IPs
Logging provides audit trail of traffic
Limitations of Firewalls
Cannot inspect encrypted traffic (without additional tools)
Cannot prevent insider threats (threats from within the network)
Cannot protect against social engineering (phishing)
Rules must be maintained — outdated rules leave gaps
A misconfigured firewall may block legitimate traffic or allow attacks
// A2.4.4 · Encryption

What is Encryption?

Encryption transforms plaintext into ciphertext using a key — making data unreadable without the correct key to decrypt it.

Plaintext
Original, readable data.

"Hello, World!"
Encryption Algorithm + Key
Mathematical function that scrambles the data using the key.

AES-256, RSA, etc.
Ciphertext
Scrambled, unreadable data — useless to an attacker without the key.

X9#k@2!mZ...

Why encryption matters: Even if an attacker intercepts data on a network, encrypted data is meaningless without the correct decryption key. This protects confidentiality during transmission.

Two main types of encryption: symmetric (same key to encrypt and decrypt) and asymmetric (pair of keys: public and private).

// A2.4.4 · Symmetric Encryption

Symmetric Encryption

The same key is used to both encrypt and decrypt data.

How it works
Sender encrypts data with Key X.
Sends encrypted data to receiver.
Receiver decrypts with the same Key X.

Example algorithms: AES (Advanced Encryption Standard), DES
Advantages & Disadvantages
✓ Fast — computationally efficient, good for large data
✓ Simple to implement
✗ Key distribution problem — how do you securely share the key in the first place?
✗ Key management — with many users, need many separate keys
// A2.4.4 · Asymmetric Encryption

Asymmetric Encryption

A pair of mathematically linked keys: a public key (shared openly) and a private key (kept secret).

Public Key
Shared publicly — anyone can have it.

Used to encrypt messages to the owner.
Can also verify a digital signature.

Like your postal address — anyone can send you a letter.
Private Key
Kept completely secret by the owner.

Used to decrypt messages encrypted with the public key.
Used to sign messages (prove identity).

Like the key to your letterbox — only you can open it.
1
Alice wants to send Bob a secret message. She gets Bob's public key (freely available).
2
Alice encrypts the message using Bob's public key. Only Bob's private key can decrypt it.
3
Alice sends the ciphertext. Even if intercepted, it's useless — only Bob's private key decrypts it.
4
Bob uses his private key to decrypt and read Alice's message.
Advantages
Solves key distribution — public key shared openly, no secret sharing needed
Enables digital signatures — proves identity
Foundation of HTTPS and secure web
Disadvantages
Much slower than symmetric encryption
Computationally expensive for large data
In practice: asymmetric used to exchange a symmetric key, then symmetric used for bulk data (TLS hybrid approach)
// A2.4.4 · Digital Certificates

Digital Certificates

A digital certificate is like a digital passport — it proves that a public key actually belongs to who they claim to be.

The Problem Without Certificates
If anyone can publish a public key, how do you know the key for "mybank.com" actually belongs to your bank, and not an attacker impersonating your bank?

This is the man-in-the-middle attack risk.
What a Certificate Contains
Owner's public key
Owner's identity (domain name, organisation)
Issuing Certificate Authority (CA)
Validity period (expiry date)
CA's digital signature (proves authenticity)
1
A trusted Certificate Authority (CA) (e.g., DigiCert, Let's Encrypt) verifies the website owner's identity.
2
The CA signs the certificate with its own private key — creating a digitally signed document that binds the domain to the public key.
3
When you visit an HTTPS site, your browser receives the certificate and verifies the CA's signature using the CA's public key (pre-installed in your browser/OS).
4
If valid, your browser shows a 🔒 padlock — you can trust the public key and establish an encrypted connection (TLS handshake).

TLS (Transport Layer Security) is the protocol that uses digital certificates + asymmetric encryption to establish a secure, encrypted HTTPS connection. SSL is its predecessor (now deprecated).

// A2.4.2 · HL — Network Vulnerabilities

Common Attacks

DDoS
Distributed Denial of Service — overwhelm a server with traffic from many compromised machines (botnet), making it unavailable to legitimate users.
MitM
Man-in-the-Middle — attacker secretly intercepts communication between two parties. Can read and modify data without either party knowing.
Phishing
Deceptive emails, messages, or websites that trick users into revealing credentials, personal data, or installing malware. Social engineering attack.
SQL Injection
Malicious SQL code inserted into an input field that is then executed by the database. Can expose, modify, or delete database contents.
XSS
Cross-Site Scripting — attacker injects malicious scripts into a trusted website. When other users load the page, the script runs in their browser.
Zero-Day
An attack exploiting a vulnerability that is unknown to the software vendor — no patch exists yet. Highly dangerous because there is no immediate defence.
Malware
Malicious software — includes viruses, ransomware, spyware, trojans. Can spread across networks, encrypting data, stealing information, or creating backdoors.
// A2.4.3 · HL — Countermeasures

Network Countermeasures

Multi-Factor Authentication (MFA)
Requires two or more verification factors: something you know (password), have (phone), or are (biometrics). Prevents account access even if password is stolen.
Encrypted Protocols
Use HTTPS instead of HTTP, SSH instead of Telnet, SFTP instead of FTP. Encrypts data in transit so MitM attackers cannot read intercepted traffic.
Input Validation
Check and sanitise all user input before processing. Prevents SQL injection and XSS. Techniques: filtering, whitelisting, parameterised queries.
IDS / IPS
Intrusion Detection System — monitors network traffic and alerts on suspicious activity.
Intrusion Prevention System — actively blocks detected threats.
DDoS Mitigation
Traffic scrubbing services filter out attack traffic before it reaches servers. Rate limiting, CDNs, and blackholing known attack sources.
Update Software
Regularly patching software closes known vulnerabilities. Defends against zero-days by reducing the attack surface over time.

Key principle: Security is multi-layered (defence in depth). No single countermeasure is sufficient — combine firewalls, encryption, MFA, input validation, and staff training.

// How HTTPS Works

The TLS Handshake

How your browser establishes a secure HTTPS connection — combining asymmetric and symmetric encryption.

1
Client Hello — Your browser sends supported TLS versions and cipher suites to the server.
2
Server Hello + Certificate — Server responds with its digital certificate (containing its public key), verified by a trusted CA.
3
Certificate Verification — Browser verifies the certificate is signed by a trusted CA and is not expired or revoked.
4
Key Exchange — Browser generates a session key (symmetric) and encrypts it with the server's public key. Sends to server.
5
Server Decrypts — Server uses its private key to decrypt and obtain the session key. Both sides now share the same symmetric key.
6
Encrypted Communication — All subsequent data is encrypted with the fast symmetric session key. Asymmetric encryption is only used for the handshake.

Why hybrid? Asymmetric encryption solves the key exchange problem securely, but is slow. Symmetric encryption is fast for bulk data. TLS uses the best of both.

// Comparison

Symmetric vs Asymmetric

FactorSymmetricAsymmetric
Keys usedOne shared keyKey pair (public + private)
SpeedFastSlow
Key distributionDifficult (must share secretly)Easy (public key shared openly)
ScalabilityPoor (n users need n keys)Good (one key pair per user)
Use casesBulk data encryption, file storageKey exchange, digital signatures, HTTPS handshake
Example algorithmsAES, DESRSA, ECC

Real-world practice: Use asymmetric to securely exchange a symmetric key. Then use the fast symmetric key for all actual communication. This is the TLS approach used in HTTPS.

// Watch Out

Common Errors

✗
Saying encryption prevents attacks — encryption protects data confidentiality but does not prevent DDoS, phishing, or SQL injection. Different threats need different countermeasures.
✗
Confusing public and private key roles — you encrypt with someone's PUBLIC key (so only they can decrypt). You sign with your own PRIVATE key (so others can verify with your public key).
✗
Saying firewalls stop all threats — firewalls are effective against external network threats but cannot prevent insider threats, social engineering, or zero-day exploits.
✗
Saying a digital certificate means a website is safe — a certificate proves identity (the site is who it claims to be), not that it is malware-free or that its content is trustworthy.
✗
Confusing SSL and TLS — SSL is the older, deprecated protocol. TLS is the current standard. In practice, "SSL certificate" is a commonly used but technically inaccurate term for TLS certificates.
