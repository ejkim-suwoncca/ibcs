// B2.4.1 · Big O Notation

Big O Notation

Big O notation describes how an algorithm's performance (time or space) scales as the input size n grows. It gives the worst-case scenario.

NotationNameDescriptionExample
O(1)ConstantTime is the same regardless of input sizeAccess array by index: nums[0]
O(log n)LogarithmicHalves the problem at each stepBinary search
O(n)LinearTime grows proportionally with inputLinear search, sum all elements
O(n²)QuadraticNested loops — time grows as n squaredBubble sort, selection sort
O(2ⁿ)ExponentialDoubles with each extra input itemSome recursive problems
Time Complexity
How many steps/operations the algorithm takes. Ignores constants and lower-order terms — only the dominant growth rate matters.
Space Complexity
How much extra memory the algorithm uses. O(1) = constant (uses same memory regardless of n). O(n) = linear extra memory needed.
// B2.4.2 · Searching

Linear Search

Check each element one by one from the start until the target is found (or the end is reached). Also called sequential search.

Algorithm Steps
Start at index 0
Compare current element to target (key)
If match → found! Return position
If no match → move to next element
If end reached → not found
Complexity
Time: O(n) — worst case checks every element

Space: O(1) — no extra memory needed

Best case: O(1) — target is at index 0

Works on unsorted lists

When to use: Small datasets (<50 items), or when the list is not sorted. Sorting a list just to binary search it may not be worth the overhead.

// B2.4.2 · Linear Search Code

Linear Search: Code & Trace

# Python — Linear Search def linear_search(numbers, key): for i in range(len(numbers)): if numbers[i] == key: return i # found at index i return -1 # not found numbers = [3, 9, 7, 4, 11] key = 7 result = linear_search(numbers, key) print("Found at index:", result) # Output: Found at index: 2

TRACE — searching for 7 in [3, 9, 7, 4, 11]:

i=0
Check numbers[0] = 3. 3 ≠ 7. Move on.
i=1
Check numbers[1] = 9. 9 ≠ 7. Move on.
i=2
Check numbers[2] = 7. 7 == 7. Found! Return 2.

In the exam: When asked to trace, show every comparison — write the index checked, the value, and whether it matched. Don't skip steps.

// B2.4.2 · Binary Search

Binary Search

Works on sorted lists only. At each step, compare the target to the middle element. Eliminate the half that can't contain the target. Repeat.

Algorithm Steps
Set lower = 0, upper = last index
Find mid = (lower + upper) // 2
If list[mid] == key → found!
If list[mid] < key → lower = mid + 1
If list[mid] > key → upper = mid − 1
Repeat until found or lower > upper
Complexity
Time: O(log n) — halves the search space each step

Space: O(1)

Requirement: List must be sorted

For 1000 items: max ~10 comparisons!

Why O(log n)? Each step halves the remaining elements. Starting with n items: after 1 step → n/2 → n/4 → ... → 1. Number of steps ≈ log₂(n).

// B2.4.2 · Binary Search Code & Trace

Binary Search: Code & Trace

# Python — Binary Search (list must be SORTED) def binary_search(numbers, key): lower = 0 upper = len(numbers) - 1 while lower <= upper: mid = (lower + upper) // 2 if numbers[mid] == key: return mid elif numbers[mid] < key: lower = mid + 1 # search right half else: upper = mid - 1 # search left half return -1 # not found numbers = [3, 4, 5, 7, 9, 11, 13] print(binary_search(numbers, 11)) # Output: 5

TRACE — searching for 11 in [3, 4, 5, 7, 9, 11, 13]:

Step 1
lower=0, upper=6, mid=3 → numbers[3]=7. 7 < 11 → lower = 4
Step 2
lower=4, upper=6, mid=5 → numbers[5]=11. 11 == 11 → Found at index 5!

Key: Only 2 comparisons for 7 elements! Linear search could take up to 7. For 1 million items, binary search takes at most 20 comparisons.

// B2.4.3 · Sorting

Bubble Sort

Compare adjacent pairs. Swap them if they're in the wrong order. Repeat passes until no swaps occur. Larger elements "bubble up" to the end.

Algorithm Steps
Outer loop: repeat n passes
Inner loop: compare adjacent elements [j] and [j+1]
If [j] > [j+1]: swap them (using temp variable)
After each pass, the last element is in its correct place
Reduce inner loop length by 1 each pass
Complexity
Time: O(n²) — two nested loops
Space: O(1) — only needs a temp variable

Simple to understand and implement
Very inefficient for large datasets

Name origin: Larger values "bubble" to the right end of the list with each pass, just like air bubbles rise to the top of water.

// B2.4.3 · Bubble Sort Trace

Bubble Sort: Code & Trace

# Python — Bubble Sort numbers = [7, 3, 2, 9] for i in range(len(numbers)): for j in range(0, len(numbers) - 1 - i): if numbers[j] > numbers[j + 1]: temp = numbers[j] numbers[j] = numbers[j + 1] numbers[j + 1] = temp print(numbers) # [2, 3, 7, 9]

TRACE — sorting [7, 3, 2, 9]:

Pass 1
Compare 7,3 → swap → [3,7,2,9]. Compare 7,2 → swap → [3,2,7,9]. Compare 7,9 → no swap. Result: [3,2,7,9] (9 is in place)
Pass 2
Compare 3,2 → swap → [2,3,7,9]. Compare 3,7 → no swap. Result: [2,3,7,9]
Pass 3
Compare 2,3 → no swap. Result: [2,3,7,9] — Sorted! ✓

Swap pattern: Always use a temp variable: temp = a[j]; a[j] = a[j+1]; a[j+1] = temp. Without temp, one value is overwritten and lost.

// B2.4.3 · Selection Sort

Selection Sort

Find the smallest element in the unsorted portion. Move it to the front of the unsorted section. Repeat.

# Python — Selection Sort numbers = [9, 7, 2, 3] for i in range(len(numbers) - 1): min_idx = i for j in range(i + 1, len(numbers)): if numbers[j] < numbers[min_idx]: min_idx = j # Swap minimum to position i temp = numbers[i] numbers[i] = numbers[min_idx] numbers[min_idx] = temp print(numbers) # [2, 3, 7, 9]

TRACE — sorting [9, 7, 2, 3]:

i=0
Find min in [9,7,2,3] → 2 at index 2. Swap 9↔2 → [2,7,9,3]
i=1
Find min in [7,9,3] → 3 at index 3. Swap 7↔3 → [2,3,9,7]
i=2
Find min in [9,7] → 7 at index 3. Swap 9↔7 → [2,3,7,9]
Done
Result: [2, 3, 7, 9] — Sorted! ✓
// B2.4.3 · Sort Comparison

Bubble vs Selection Sort

PropertyBubble SortSelection Sort
Time ComplexityO(n²)O(n²)
Space ComplexityO(1)O(1)
StrategyCompare adjacent pairs, swap if out of orderFind minimum, move it to sorted section
Number of swapsMany (one per comparison)Fewer (one per pass)
Early exit possible?Yes (with "swapped" flag — if no swaps in a pass, already sorted)No — always completes all passes
Best caseO(n) with early exit (already sorted)O(n²) always

Both are O(n²): Neither is efficient for large datasets. But selection sort makes fewer swaps, which can matter when swapping is expensive. Bubble sort with a swapped flag can exit early on nearly-sorted data.

Key insight: Binary search is far more efficient than linear for large sorted datasets. But the data must be sorted first — and sorting costs O(n²) with these basic algorithms, so weigh the trade-off based on how often you need to search.

// B2.4 · Summary

Key Takeaways

Big O Notation
Describes worst-case growth of time/space
O(1) constant < O(log n) < O(n) < O(n²)
Ignore constants and lower-order terms
Two nested loops = O(n²)
Search Algorithms
Linear — O(n), works on any list
Binary — O(log n), requires sorted list
Use binary for large sorted data
Use linear for small or unsorted data
Sort Algorithms
Bubble — compare adjacent, swap if wrong
Selection — find min, place it at front
Both O(n²) time, O(1) space
Selection makes fewer swaps
