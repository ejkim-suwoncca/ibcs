B1.1.1 · Problem Specification 02 / 11

Understand the problem before you solve it.

A problem specification answers six questions. Miss one, and you risk building the wrong thing.

Problem Statement
Who has the problem? What is going wrong? What needs to change?
Constraints
What limits the solution? E.g. budget, time, hardware, laws.
Objectives
What must the solution do? Agreed with the stakeholders.
Inputs & Outputs
What data goes in? What is produced? In what format?
Success Criteria
Specific, measurable tests that tell you the solution works.
Stakeholders
People who use or are affected by the solution. Interview them first.
IA link: Your success criteria here are reused at the end of your project to evaluate your final product.
B1.1.1 · Inputs & Outputs 03 / 11

How data enters and leaves the system.

You must identify both before you can design an algorithm.

Input methods
Manual entry
Keyboard, mouse, touchscreen — typed or clicked by a person. Simple but error-prone.
Direct entry
Barcode scanner, OMR, OCR — reads data automatically from a source.
Sensor / automatic
Temperature, motion, light sensors. No human needed.
Output types
Temporary (soft copy)
On a screen. Disappears when the program ends.
Permanent (hard copy)
Printed on paper. A physical, lasting record.
Mechanical / actuator
Motors or relays that move something in the real world.
Exam tip: Always state both the type and the format of inputs/outputs (e.g. "integer entered via keyboard").
B1.1.2 · The CT Process 04 / 11

Seven steps from problem to solution.

Computational thinking is a process, not a single skill. Each step feeds into the next.

#StageWhat you do
1Real-world problemIdentify a situation that needs a computational solution.
2Problem specificationDefine inputs, outputs, constraints, objectives and success criteria.
3DecompositionBreak the big problem into smaller, manageable sub-problems.
4Pattern recognitionFind similarities or repetition across sub-problems.
5AbstractionKeep only the details that matter; ignore the rest.
6Algorithmic thinkingDesign a clear, ordered set of steps to solve each part.
7Testing & evaluationCheck correctness against your success criteria.
Key point: Steps 3–6 are the four pillars of computational thinking. You need to know each one by name and definition for the exam.
B1.1.2 · Decomposition 05 / 11

Break a big problem into smaller parts.

Each smaller part is easier to understand, design, and test on its own. Then combine the parts into a full solution.

Why decompose?
  • Easier to understand each part
  • Different people can work on different parts
  • Easier to test one part at a time
  • Easier to find and fix bugs
Important: Decomposition splits the problem, not the code. You do this before writing a single line.
Example — school library system
Full problem ↓ broken into parts
📚 Search for a book
🔐 Log in / log out
📋 Borrow & return books
🔔 Send overdue reminders
B1.1.2 · Pattern Recognition & Abstraction 06 / 11

Find what repeats. Remove what doesn't matter.

Pattern recognition
Spot similarities across sub-problems so you can reuse the same solution instead of rewriting it.
Example: "Borrow a book" and "Return a book" both need to check the member ID. Write that check once, use it twice.
Payoff
Shorter code · fewer bugs · easier to update
Abstraction
A simplified model of something complex. Keep only the details needed for the task; hide everything else.
Real-world example: A bus timetable is an abstraction of the full bus system. It shows you departure times — not the engine specs, driver names, or fuel level.
In code: A function is an abstraction — you call it without knowing exactly how it works inside.
B1.1.2 · Algorithmic Thinking 07 / 11

Write a precise, step-by-step solution.

An algorithm is a finite, ordered sequence of unambiguous steps that solves a problem. Every step must be clear enough that a computer can follow it exactly.

Three rules for a good algorithm
  1. Finite — it must eventually stop.
  2. Ordered — sequence matters; wrong order, wrong result.
  3. Unambiguous — each step has exactly one meaning.
Common mistake: Writing steps in the wrong order produces the wrong answer even when every individual step is correct.
Pseudocode example — find the larger number
INPUT a INPUT b IF a > b THEN OUTPUT "Larger is ", a ELSE OUTPUT "Larger is ", b END IF
Pseudocode is plain-English code used to plan algorithms — the IB uses its own style. Learn the IB keywords: INPUT, OUTPUT, IF/THEN/ELSE, LOOP, etc.
B1.1.4 · Flowchart Symbols 08 / 11

Five symbols — memorise them all.

START / END
Oval / Terminator
Marks the beginning or end of the algorithm.
INPUT / OUTPUT
Parallelogram
Reading input from the user or displaying output.
PROCESS
Rectangle
A calculation or variable assignment.
DECISION
Diamond
A yes/no question. Always label both branches (YES / NO).
↓
Arrow / Flowline
Shows the direction the algorithm flows.
Example flowchart — average of two numbers
START
↓
INPUT a, b
↓
total ← a + b
↓
avg ← total ÷ 2
↓
OUTPUT avg
↓
END
Exam tip: A flowchart question will ask you to draw or complete a diagram. Use the correct symbol for each step — wrong symbol = lost marks.
B1.1.4 · Trace Tables 09 / 11

Check your algorithm, step by step.

A trace table manually tracks every variable through each line of an algorithm. Used to find logic errors before running any code.

Algorithm — find the larger number
1. INPUT a → a = 7 2. INPUT b → b = −3 3. IF a > b THEN 4. max ← a 5. ELSE 6. max ← b 7. END IF 8. OUTPUT max
How to fill a trace table:
After every line, write each variable's current value. If a variable didn't change, copy its last value — don't leave it blank.

B1.1.3 · CT in Practice 10 / 11

The same four skills, everywhere.

Decomposition, pattern recognition, abstraction, and algorithmic thinking appear across all areas of computer science — the labels just change.

Software development
Decompose into modules → find patterns → abstract shared code → test algorithms.
Data analysis
Pattern recognition reveals trends. Abstraction filters noise. Algorithms process the data.
Database design
Decompose data into tables. Abstract away complexity with queries. Algorithms sort and search.
Machine learning / AI
Patterns drive the model. Abstraction selects the right features. Testing checks accuracy.
Think about it: Can you identify all four CT skills in your IA project? Examiners reward students who name and apply them explicitly.
