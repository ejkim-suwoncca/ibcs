// Foundation

What is a Database?

A database is an organised collection of structured data stored electronically so it can be quickly retrieved and manipulated.

Without a Database
Data stored in spreadsheets or flat files

Same data duplicated across many files

Hard to search, update, or ensure consistency

No way to enforce rules or relationships
With a Relational Database
Data split into related tables

No duplication — one source of truth

Fast queries using SQL

Relationships enforced by keys

A Relational Database Management System (RDBMS) is the software that stores and manages the database — e.g. MySQL, SQLite, PostgreSQL, Microsoft SQL Server.

// Feature 1 · Tables

Tables: Structure of Data

A table stores data about a single entity (a person, product, flight, etc.) in rows and columns.

Entity
A real-world thing we store data about

Example: Student, Course, Teacher
Attribute (Column)
A characteristic of the entity

Example: StudentID, FirstName, Email
Tuple / Record (Row)
One instance of the entity

Example: one specific student's data

EXAMPLE TABLE: STUDENT

StudentIDFirstNameLastNameDateOfBirthEmail
S001EJKim2009-03-14ej.kim@suwoncca.org
S002JiyeonPark2009-07-22jiyeon.park@suwoncca.org
S003MinjunLee2010-01-05minjun.lee@suwoncca.org

StudentID highlighted in gold = Primary Key — each value is unique and identifies exactly one student.

// Feature 2 · Keys

Primary Key

A primary key is an attribute (or set of attributes) that uniquely identifies every record in a table. No two rows can have the same primary key value.

Simple Primary Key
A single attribute that is unique

Example: StudentID = "S001"

Auto-generated numbers are common: 1, 2, 3…
Composite Key
Two or more attributes combined to form a unique identifier

Example: FlightID + PilotID in a PilotFlight table

Neither alone is unique, but together they are
Rules for Primary Keys
Unique — no duplicate values
Not null — must always have a value
Stable — should not change over time

Common mistake: Using a person's name or email as a primary key is risky — names can be duplicated, and emails can change. Always prefer a stable, system-generated ID.

// Feature 3 · Foreign Key

Foreign Key

A foreign key is an attribute in one table that refers to the primary key in another table. It creates a link between two tables.

TABLE: STUDENT

StudentID (PK)FirstNameLastName
S001EJKim
S002JiyeonPark

TABLE: ENROLMENT

EnrolmentID (PK)StudentID (FK)CourseCode
E01S001IBCS-SL
E02S001IBMATH-SL
E03S002IBCS-SL

Referential Integrity: The database will reject any foreign key value that doesn't match an existing primary key. You cannot enrol a student who doesn't exist in the STUDENT table.

Gold (PK): StudentID in STUDENT table. Blue (FK): StudentID in ENROLMENT — it references the primary key in STUDENT.

// Feature 4 · Relationships

Table Relationships

Relationships define how records in one table connect to records in another. There are three types:

1 : 1
One-to-One — one record in Table A links to exactly one record in Table B.
Example: one person has one passport number.
1 : M
One-to-Many — one record in Table A links to many records in Table B (most common).
Example: one student enrols in many courses. One teacher teaches many students.
M : M
Many-to-Many — many records in Table A link to many in Table B. Requires a linking table.
Example: many students take many courses. Solved with an ENROLMENT junction table.

Why can't we store M:M directly? A foreign key holds a single value — it can't reference multiple rows. A junction (linking) table solves this by breaking M:M into two 1:M relationships.

// A3.1.1 · Benefits

Benefits of Relational Databases

Reduced Redundancy
Data is stored only once. Normalisation removes repeated groups. Updates only need to happen in one place — no inconsistency.
Data Integrity
Referential integrity rules prevent orphaned records. Constraints ensure data is valid and consistent across all tables.
Data Security
Access controls restrict who can view, edit, or delete data. Views can hide sensitive columns from certain users.
Concurrency Control
Many users can access the database simultaneously. Transaction management prevents data corruption from simultaneous writes.
Scalability
Relational databases scale vertically — add more CPU/RAM. Handles growing datasets and more users without loss of performance.
Community Support
Relational databases have existed since the 1970s. Huge online communities, documentation, and tools available for troubleshooting.
// A3.1.1 · Limitations

Limitations of Relational Databases

Big Data Scalability
Relational databases scale vertically (better hardware), not horizontally (more machines). At massive scale (billions of rows), they struggle compared to NoSQL alternatives.
Design Complexity
Designing a well-normalised schema requires significant planning. Changes to the schema later (e.g., adding tables) can be complex and risky.
Rigid Schema
Every record must conform to the table structure. If your data is irregular or evolving, a rigid schema becomes a problem.
Unstructured Data
Cannot efficiently store unstructured data (images, free-form text, JSON). Purpose-built databases (NoSQL, graph) are better for this.
Object-Relational Mismatch
OOP languages use objects/inheritance — relational databases use tables/rows. Mapping between them requires extra code (ORM frameworks).
Hierarchical Data
Representing parent-child hierarchies (like a file system or org chart) requires complex joins. Graph databases handle this more naturally.
// Summary · Key Types

Types of Keys

Key Type Definition Example
Primary Key Uniquely identifies every record in a table. Cannot be null or duplicated. StudentID = "S001"
Foreign Key References the primary key of another table to create a relationship. StudentID in ENROLMENT references STUDENT
Composite Key Two or more attributes combined to form a unique identifier. FlightID + PilotID in PILOTFLIGHT table

Exam tip: When explaining keys in a scenario, always state why the chosen attribute uniquely identifies the record — and explain what would happen without referential integrity.

// Data Integrity

Reliable Transactions

A transaction is a sequence of database operations treated as a single unit — either all succeed, or none do.

Example: Bank Transfer
1. Deduct ₩50,000 from Account A
2. Add ₩50,000 to Account B

If step 2 fails, step 1 must be rolled back.
Without this, money disappears.
Why Transactions Matter
Ensures data integrity during failures

Prevents partial updates from corrupting the database

Covered in detail in A3.3 (ACID properties)

Common mistake: When asked about benefits in a scenario, you must explain how they apply to that specific context — not just list them generically. Connect each benefit to the scenario given.

// Exam Strategy

How to Answer These Questions

Q
"A hospital uses a relational database to store patient records. Explain one benefit of using a relational database in this scenario." [2 marks]
✓
Good answer: "Data integrity is maintained — the foreign key in the APPOINTMENT table must reference a valid PatientID in the PATIENT table, preventing appointments being linked to non-existent patients, which could be dangerous in a medical context."
✗
Weak answer: "Data integrity is maintained. This is good for the hospital." — too vague, no connection to the scenario, will not score full marks.

Formula: State the benefit → Explain how it works → Apply it to the given scenario = full marks.

Key Takeaways

Core Vocabulary

Attribute — column (one property)
Tuple/Record — one row (one instance)
Entity — a thing we store data about
Keys

Composite Key — two+ attributes combined

Relationships
1:1 — one-to-one (rare)
1:M — one-to-many (most common)
M:M — many-to-many → needs junction table
