// A3.3.1 · DDL vs DML

DDL vs DML

SQL has two main categories of commands:

DDL — Data Definition Language
Defines and modifies the structure of the database

Commands: CREATE, ALTER, DROP

Examples: create a table, add a column, delete a table

Think: building the blueprint
DML — Data Manipulation Language
Adds, modifies, deletes, and retrieves the data inside the database

Commands: SELECT, INSERT, UPDATE, DELETE

Examples: find all students in Year 11, update a price, delete an old order

Think: filling in and using the blueprint
CommandCategoryPurpose
CREATE TABLEDDLCreate a new table with defined columns and types
ALTER TABLEDDLModify an existing table (add/remove columns, constraints)
DROP TABLEDDLDelete a table and all its data permanently
SELECTDMLRetrieve data from one or more tables
INSERT INTODMLAdd new records to a table
UPDATE ... SETDMLModify existing records
DELETE FROMDMLRemove records from a table
// DDL · CREATE TABLE

Creating Tables

The CREATE TABLE statement defines a new table, its columns, data types, and constraints.

CREATE TABLE Students ( StudentID INTEGER PRIMARY KEY, FirstName VARCHAR(50) NOT NULL, LastName VARCHAR(50) NOT NULL, DateOfBirth DATE, Email VARCHAR(100) UNIQUE );
CREATE TABLE Enrolments ( EnrolID INTEGER PRIMARY KEY, StudentID INTEGER NOT NULL, CourseCode VARCHAR(20) NOT NULL, EnrolDate DATE, FOREIGN KEY (StudentID) REFERENCES Students(StudentID), FOREIGN KEY (CourseCode) REFERENCES Courses(CourseCode) );

NOT NULL — this field must have a value. UNIQUE — no two rows can share the same value. FOREIGN KEY — enforces referential integrity with another table.

// DML · SELECT Basics

Querying with SELECT

SELECT retrieves data from a table. It is the most used SQL command.

-- Retrieve all columns and all rows SELECT * FROM Students;
-- Retrieve specific columns only SELECT FirstName, LastName, Email FROM Students;
-- Filter with WHERE — only students in year 11 SELECT FirstName, LastName FROM Students WHERE YearGroup = 11;
-- Sort results alphabetically by last name SELECT FirstName, LastName FROM Students ORDER BY LastName ASC;

Syntax structure: SELECT [columns] FROM [table] WHERE [condition] ORDER BY [column] ASC/DESC;

// DML · Filtering

Filtering with WHERE

-- Relational operators SELECT * FROM Students WHERE Age > 16; SELECT * FROM Students WHERE Age <= 18; SELECT * FROM Students WHERE YearGroup != 10;
-- BETWEEN filters a range (inclusive) SELECT * FROM Employees WHERE Salary BETWEEN 50000 AND 100000;
-- LIKE for pattern matching -- % = any number of characters, _ = exactly one character SELECT * FROM Students WHERE LastName LIKE 'K%'; -- starts with K SELECT * FROM Students WHERE Email LIKE '%@suwoncca.org';
-- Combining conditions with AND / OR SELECT * FROM Students WHERE YearGroup = 11 AND LastName LIKE 'K%';
-- DISTINCT — remove duplicate values SELECT DISTINCT YearGroup FROM Students;
// A3.3.2 · JOIN

Joining Tables

A JOIN combines rows from two tables based on a related column (usually a foreign key linking to a primary key).

-- Get student names with their enrolled course codes SELECT Students.FirstName, Students.LastName, Enrolments.CourseCode FROM Students JOIN Enrolments ON Students.StudentID = Enrolments.StudentID;
-- Three-table join: get student name + course title SELECT Students.FirstName, Students.LastName, Courses.Title FROM Students JOIN Enrolments ON Students.StudentID = Enrolments.StudentID JOIN Courses ON Enrolments.CourseCode = Courses.CourseCode;
-- Join with WHERE filter + ORDER BY SELECT Students.LastName, Courses.Title FROM Students JOIN Enrolments ON Students.StudentID = Enrolments.StudentID JOIN Courses ON Enrolments.CourseCode = Courses.CourseCode WHERE Courses.Level = 'SL' ORDER BY Students.LastName ASC;

ON specifies the condition that links the two tables — always a PK = FK relationship. Without it, you get a Cartesian product (every row combined with every other row).

// DML · Grouping & Ordering

GROUP BY & HAVING

GROUP BY groups rows with the same value. HAVING filters groups (like WHERE, but for groups after aggregation).

-- Count how many students are in each year group SELECT YearGroup, COUNT(*) AS NumStudents FROM Students GROUP BY YearGroup;
-- Only show year groups with more than 20 students SELECT YearGroup, COUNT(*) AS NumStudents FROM Students GROUP BY YearGroup HAVING COUNT(*) > 20;
-- Average salary by department (with JOIN) SELECT Departments.Name, AVG(Employees.Salary) AS AvgSalary FROM Employees JOIN Departments ON Employees.DeptID = Departments.DeptID GROUP BY Departments.Name ORDER BY AvgSalary DESC;

WHERE vs HAVING: WHERE filters rows before grouping. HAVING filters groups after aggregation. You cannot use HAVING without GROUP BY.

// A3.3.3 · DML · INSERT

Adding Data: INSERT INTO

INSERT INTO adds new records to a table.

-- Insert a single record (specify columns) INSERT INTO Students (StudentID, FirstName, LastName, Email) VALUES (1001, 'EJ', 'Kim', 'ej.kim@suwoncca.org');
-- Insert multiple records at once INSERT INTO Students (StudentID, FirstName, LastName) VALUES (1002, 'Jiyeon', 'Park'), (1003, 'Minjun', 'Lee'), (1004, 'Sora', 'Choi');

Column order matters: Values must match the order of the columns listed. If a column has a NOT NULL constraint and is not listed, the insert will fail.

Common error: Inserting a foreign key value that doesn't exist in the referenced table will fail (referential integrity violation). Always insert parent records first.

// A3.3.3 · DML · UPDATE

Modifying Data: UPDATE

UPDATE modifies existing records. Always use WHERE to target specific rows.

-- Update one student's email UPDATE Students SET Email = 'ej.kim2@suwoncca.org' WHERE StudentID = 1001;
-- Update multiple columns at once UPDATE Employees SET Salary = 55000, Department = 'Engineering' WHERE EmployeeID = 42;
-- Give all Year 11 students a 5% grade bonus UPDATE Students SET GradeScore = GradeScore * 1.05 WHERE YearGroup = 11;

⚠️ Critical warning: Omitting the WHERE clause updates every row in the table. Always double-check your WHERE condition before executing an UPDATE.

// A3.3.3 · DML · DELETE

Removing Data: DELETE

DELETE FROM removes records. Like UPDATE, always use WHERE — or all rows are deleted.

-- Delete a specific student record DELETE FROM Students WHERE StudentID = 1004;
-- Delete all students who graduated (year > current) DELETE FROM Students WHERE GraduationYear < 2026;
-- DANGER: Delete ALL rows (no WHERE = all deleted!) DELETE FROM Students; -- Deletes every student!
Referential Integrity Danger
If a student has enrolment records in ENROLMENTS, deleting them from STUDENTS may fail — the FK constraint prevents orphaned records.
Solution: Delete in Order
Delete child records first (ENROLMENTS), then delete the parent (STUDENTS). Or use CASCADE DELETE in the table definition.

// Common Errors

Common SQL Mistakes

✗
Forgetting WHERE on UPDATE/DELETE — without WHERE, all rows are affected. Always specify which rows you want to change.
✗
Using WHERE instead of HAVING — WHERE cannot filter aggregate functions (COUNT, SUM, AVG). Use HAVING after GROUP BY for that.
✗
JOIN without ON — always specify the linking condition. A JOIN without ON creates a massive Cartesian product.
✗
Inserting child before parent — if ENROLMENTS has a FK to STUDENTS, you must insert the student first, then the enrolment.
✗
Using single quotes for column names — strings use single quotes ('EJ'). Column and table names use no quotes or backticks. Never mix them up.
// A3.3 · Summary

Key Takeaways

DDL — Structure
CREATE TABLE — define structure & types
ALTER TABLE — modify existing structure
DROP TABLE — delete table permanently
FOREIGN KEY enforces referential integrity
DML — Data
SELECT — retrieve data
INSERT INTO — add new records
UPDATE … SET — modify records
DELETE FROM — remove records
Querying Power
JOIN … ON — combine tables
WHERE — filter rows
GROUP BY / HAVING — aggregate
LIKE / BETWEEN — pattern & range
Key Rules
Always use WHERE with UPDATE/DELETE
Use HAVING (not WHERE) after GROUP BY
Insert parent records before child records
Single quotes = string values
