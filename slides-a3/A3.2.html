// A3.2.1 · Schemas

Three Levels of Schema

A database schema is a blueprint describing how data is organised and how relationships are managed. There are three levels:

1. Conceptual Schema
What: High-level, abstract view

Lists entities and their relationships — no data types, no implementation details

Audience: Business stakeholders, non-technical team
2. Logical Schema
What: Detailed design

Defines tables, attributes, data types, keys, and relationships — but not DBMS-specific

Audience: Database designers
3. Physical Schema
What: Implementation details

Specifies storage, indexes, partitioning, for a specific DBMS (e.g. MySQL)

Audience: Database administrators (DBAs)

Analogy: Conceptual = "we need a library with books and members." Logical = the actual table structures and types. Physical = "indexed by LastName, stored on SSD partition 2."

// A3.2.1 · Schema Example

Schema Levels: Sales System

Conceptual
Entities: Customers, Orders, Products

Relationships:
Customer places Order
Order includes Product(s)
Logical
CUSTOMERS(CustomerID: INT PK, Name: VARCHAR, Email: VARCHAR)
ORDERS(OrderID: INT PK, Date: DATE, CustomerID: INT FK)
PRODUCTS(ProductID: INT PK, Name: VARCHAR, Price: REAL)
Physical
CustomerID INT AUTO_INCREMENT PRIMARY KEY
Name VARCHAR(100) NOT NULL
INDEX on Email for fast lookup
Partition ORDERS by OrderDate

Why three levels? Separating concerns allows designers, developers, and DBAs to each work at their appropriate level without needing to understand all the others.

// A3.2.2 · Entity-Relationship Diagrams

What is an ERD?

An Entity-Relationship Diagram (ERD) is a visual representation of the entities in a database and the relationships between them.

Purpose of ERDs
Plan the database before coding

Communicate the design to stakeholders

Document the structure for future reference

Identify missing entities or relationships early
ERD Symbols (Chen notation)
Rectangle = Entity (table)
Ellipse/oval = Attribute (column)
Diamond = Relationship
Lines = connections with cardinality markers

IB style: For this course, you need to construct ERDs showing entities, attributes, and relationships with cardinality (1:1, 1:M, M:M) and modality (optional vs mandatory).

// A3.2.2 · ERD Concepts

Cardinality & Modality

Cardinality
The maximum number of instances of one entity that can be associated with another.

1 = exactly one
M or N = many

Types: 1:1, 1:M, M:M
Modality
The minimum number — whether participation is optional or mandatory.

0 = optional (entity can exist without the relationship)
1 = mandatory (must be related to at least one)
ScenarioCardinalityModality
Patient has medical records1:M (one patient, many records)Patient: optional (new patients may have none)
Order belongs to CustomerM:1 (many orders, one customer)Order: mandatory (every order must have a customer)
Student enrols in CourseM:M (many students, many courses)Student: optional; Course: optional

// A3.2.3 · Data Types

Database Data Types

Each attribute must be assigned the correct data type. Choosing the wrong type causes storage waste, errors, and prevents valid operations.

Data TypeDescriptionExample use
INTEGERWhole number — no decimal partStudentID, Age, Quantity
REAL / FLOATNumber with a decimal partPrice, GPA, Weight
VARCHAR(n)Variable length text — max n charactersName, Email, Description
CHARACTER(n)Fixed-length text — always n charactersCourseCode, CountryCode
DATEDate stored as YYYY-MM-DDDateOfBirth, OrderDate
TIMETime stored as HH:MM:SSDepartureTime, LoginTime
BOOLEANTrue or False onlyIsActive, HasPaid, IsAdmin

Why does data type matter? Using VARCHAR for a price prevents arithmetic (you can't SUM text). Using INTEGER for a phone number loses leading zeros. Always match the type to the real-world meaning of the data.

// A3.2.5 · Normalisation

What is Normalisation?

Normalisation is the process of organising data in a relational database to reduce redundancy and improve data integrity.

Problems Without Normalisation
Insertion anomaly — can't add a course without a student

Update anomaly — changing a teacher's name requires updating many rows

Deletion anomaly — deleting last enrolment may delete course data
Goals of Normalisation
Each piece of data stored in one place only

Each table represents one entity

Non-key attributes depend only on the primary key

There are multiple Normal Forms (NF). The IB requires you to understand 1NF, 2NF, and 3NF. Each builds on the previous.

// A3.2.5 · First Normal Form

1NF — First Normal Form

A table is in 1NF when:

①
All attribute values are atomic — each cell holds one indivisible value (no lists, no arrays)
②
There are no repeating groups of attributes
③
A primary key is defined to uniquely identify each row

✗ NOT in 1NF

StudentIDNameClubs
S001EJ KimRobotics, Chess
S002JiyeonChess

✓ In 1NF

StudentIDNameClub
S001EJ KimRobotics
S001EJ KimChess
S002JiyeonChess

Key term — Atomic: A value that cannot be broken down further. "EJ Kim" is not atomic — it should be split into FirstName and LastName.

// A3.2.5 · Second Normal Form

2NF — Second Normal Form

A table is in 2NF when it is in 1NF AND every non-key attribute is fully functionally dependent on the entire primary key (not just part of it).

2NF only matters when using a composite primary key. If the PK is a single attribute, 1NF → 2NF is automatic.

✗ NOT in 2NF (partial dependency)

OrderID (PK)ProductID (PK)QtyProductName
O01P012USB-C Hub
O01P021Keyboard

ProductName only depends on ProductID — not the full composite key

✓ In 2NF — split into two tables

OrderID (PK)ProductID (PK)Qty
O01P012
ProductID (PK)ProductName
P01USB-C Hub
P02Keyboard
// A3.2.5 · Third Normal Form

3NF — Third Normal Form

A table is in 3NF when it is in 2NF AND there are no transitive dependencies — non-key attributes must not depend on other non-key attributes.

Transitive dependency: A → B → C, where A is the PK but B (a non-key) determines C. C is transitively dependent on A through B.

✗ NOT in 3NF (transitive dependency)

StudentID (PK)NamePostalCodeCity
S001EJ Kim16488Suwon
S002Jiyeon03925Seoul

City depends on PostalCode, not on StudentID directly

✓ In 3NF — split the transitive dependency

StudentID (PK)NamePostalCode (FK)
S001EJ Kim16488
PostalCode (PK)City
16488Suwon
03925Seoul
// Summary · Normal Forms

Normal Forms at a Glance

Normal FormRequirementFixes
1NF Atomic values only. No repeating groups. Has a primary key. Multi-valued attributes, lists in cells
2NF In 1NF + no partial dependencies on composite key. Attributes depending on only part of the PK
3NF In 2NF + no transitive dependencies (non-key → non-key). Non-key attributes depending on other non-key attributes

Mnemonic: "The key, the whole key, and nothing but the key" — 1NF (the key), 2NF (the whole key), 3NF (nothing but the key).

// A3.2.7 · Denormalisation

When to Denormalise?

Denormalisation intentionally introduces redundancy to improve read performance, at the cost of some data integrity.

Advantages of Denormalisation
Faster reads — fewer JOIN operations needed

Simpler queries — all data in one table

Useful for read-heavy systems (reporting dashboards, analytics)
Disadvantages
Data redundancy — same data stored multiple times

Update anomalies — must update data in multiple places

Slower writes — inserts and updates are more complex

When to denormalise: In reporting/analytical databases where queries are run constantly and speed matters more than write performance. Not appropriate for transactional systems where data must always be accurate.

// Common Errors

Common Mistakes

✗
Confusing modality and cardinality — Cardinality = how many (max). Modality = whether it's required (min: 0 or 1).
✗
Thinking 2NF applies to all tables — 2NF only matters with composite primary keys. If PK is a single column, 1NF already implies 2NF.
✗
Stating normalisation benefits generically — always link to the scenario. "Reducing redundancy means the teacher's name only needs updating once, not in every student record."
✗
Using the wrong data type — Phone numbers look like integers but should be VARCHAR (leading zeros, formatting symbols). Match data types to real-world meaning.
