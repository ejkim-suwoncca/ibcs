// A3.4.1 · NoSQL Databases

Beyond Relational — NoSQL

Relational databases organize data in rigid tables with fixed schemas. NoSQL databases offer flexible, scalable alternatives designed for large volumes of diverse data.

Why NoSQL?
Flexibility: No fixed schema — store unstructured or semi-structured data
Scalability: Designed to scale horizontally across many servers
Speed: Optimized for fast read/write at massive scale
Variety: Different models for different data shapes
Relational vs NoSQL
Relational: Tables, fixed schema, SQL, ACID, great for structured data with many relationships

NoSQL: Flexible schema, various models, eventual/strong consistency, great for big data, real-time apps, and varied data shapes
①
Document — data as JSON-like objects (e.g. MongoDB)
②
Key-Value — simple key → value pairs (e.g. Redis)
③
Wide-column — tables with dynamic columns per row (e.g. Cassandra)
④
Graph — nodes and edges for relationship-heavy data (e.g. OrientDB)
// A3.4.1 · Four NoSQL Types

Type Structure Example DB Best For Trade-off

// A3.4.1 · Specialty Databases

Specialty Database Types

☁️ Cloud Databases
Run on cloud platforms (AWS, Azure, GCP). Two models:

Self-managed (IaaS): organization controls deployment but cloud provides infrastructure

Managed (DBaaS): vendor handles everything — provisioning, backups, scaling, patches

Examples: Amazon DynamoDB, Azure Cosmos DB, Amazon RDS
📍 Spatial Databases
Optimized for data related to objects in space — points, lines, polygons, 3D coordinates.

Supports spatial data types, spatial indexes, and geometric functions.

Used for: GIS, mapping apps, location-based services

Examples: PostGIS, Oracle Spatial, MongoDB Geospatial
⚡ In-Memory Databases
Store all data entirely in RAM rather than on disk — providing extremely fast read/write operations.

Used for: real-time analytics, caching, gaming leaderboards

Trade-off: data lost if power fails (low durability without persistence)

Examples: Redis, Oracle TimesTen

Exam tip: Know which database type fits which scenario. Key factors: data structure (structured/semi-structured/unstructured), query complexity, speed requirements, scale, and whether relationships are important.

// A3.4.2 · Data Warehouses

What is a Data Warehouse?

A data warehouse is a centralized repository that consolidates large volumes of data from multiple sources for historical analysis and business intelligence — not for day-to-day transactions.

Operational DB (OLTP) vs Data Warehouse (OLAP)
Operational (OLTP):
• Used for daily transactions (INSERT, UPDATE)
• Current data only
• Many small, fast reads/writes
• e.g. school database, bank records

Data Warehouse (OLAP):
• Used for analytical queries and reporting
• Historical data over time
• Few large, complex queries
• e.g. "What were sales trends last 5 years?"
Key Characteristics
Append-only: Existing records aren't updated — new rows are added to preserve history

Subject-oriented: Organized around topics (sales, finance, marketing)

Integrated: Data from multiple source systems, unified via ETL

Time-variant: Tracks data changes over time for trend analysis

Non-volatile: Once loaded, data is rarely updated or deleted
// A3.4.2 · ETL Process

ETL — Extract, Transform, Load

ETL is the process used to move data from multiple operational sources into the data warehouse. It ensures data is clean, consistent, and in the right format for analysis.

E
Extract — Gather raw data from various operational databases, flat files, APIs, and external sources. Must not significantly impact the source system's performance.
T
Transform — Clean, validate, and reformat the data. Resolve inconsistencies (e.g. "USA" vs "United States"), aggregate data, convert data types, and apply business rules.
L
Load — Write the transformed data into the data warehouse. Can be a full load (all data) or incremental load (only new/changed records since last load).
ETL Precaution 1
Extraction must not affect the performance of the source system during peak hours
ETL Precaution 2
Different data formats must be converted to a consistent format the warehouse can read
ETL Precaution 3
Only relevant data for the intended analysis should be extracted — not everything
// A3.4.3 · OLAP

OLAP — Online Analytical Processing

OLAP software enables users to analyze data from multiple dimensions simultaneously — such as time, product, and region — to support complex business decisions.

What OLAP Does
• Provides pre-aggregated views of data optimized for querying
• Supports complex calculations on aggregated data (e.g. year-over-year comparisons)
• Enables ad hoc queries — users can explore data dynamically
• Supports data visualization: charts, dashboards, reports
• Answers questions like: "How did Q3 Korea sales compare to Q3 Japan sales across the last 4 years?"
OLAP Cube — The Mental Model
Think of data organized as a 3D cube where each axis is a dimension:

Axis 1: Time (Year → Quarter → Month)
Axis 2: Product (Category → Subcategory → Item)
Axis 3: Region (Country → City → Store)

You can slice (fix one dimension), dice (select a sub-cube), or drill down/up (change granularity) to explore the data.

OLAP vs OLTP: OLAP = analytical, few complex queries, historical data, read-heavy. OLTP = transactional, many simple queries, current data, read/write-heavy.

// A3.4.3 · Data Mining

Data Mining

Data mining is the process of discovering hidden patterns and relationships in large datasets using statistical algorithms, machine learning, and AI — to extract actionable business insights.

What Data Mining Finds
Patterns: "Customers who buy X tend to also buy Y"
Predictions: Forecast future demand, customer churn, or fraud risk based on historical data
Segmentation: Group customers by behavior for targeted marketing
Anomalies: Detect unusual transactions that may indicate fraud or errors
Real-World Examples
🛒 Amazon: "Customers who bought this also bought…" — association rules

💳 Banks: Fraud detection — transaction anomalies in real time

🏥 Healthcare: Predict patient readmission risk from medical history

📧 Marketing: Customer segmentation for personalized email campaigns

Business intelligence = OLAP (explore aggregated data) + Data Mining (discover hidden patterns) + Visualization (present findings). Together they transform raw data into informed decisions.

// A3.4.4 · Distributed Databases

Distributed Databases

A distributed database is a database whose data is stored across two or more sites on the same or different networks — but presents a unified, transparent view to users and applications.

Why Distribute?
Scalability: Add more nodes as data grows instead of upgrading one server
Availability: If one node fails, others continue serving requests
Locality: Store data closer to where it is accessed (e.g. Korea data on Asian servers)
Performance: Distribute query load across multiple servers
Real-World Users
🛍 Online retailers — manage product catalogues and inventory across distributed warehouses globally

📡 Telecom companies — manage subscriptions and network traffic across different geographical locations

🌐 Social media platforms — billions of users, data stored in data centers worldwide

Key challenge: Maintaining data consistency across all nodes when multiple users simultaneously update the same data from different locations.

// A3.4.4 · ACID in Distributed Systems

ACID Across Multiple Nodes

ACID properties (from A3.3) become more complex in distributed systems because a single transaction may span multiple nodes across a network.

A
Atomicity — distributed transactions are either committed across all nodes or rolled back completely. No partial updates across nodes — prevents inconsistency.
C
Consistency — after a transaction completes, all nodes must have access to the same consistent view of the data. No node should see stale or conflicting information.
I
Isolation — concurrent transactions on different nodes cannot interfere with each other. Prevents two transactions from modifying the same data simultaneously in conflicting ways.
D
Durability — committed transactions are reliably stored and replicated across distributed nodes. Even if a node fails after committing, the data is not lost.

Example: Two systems attempt to update the same bank balance simultaneously. The solution is to lock the transaction — one waits until the other commits, then proceeds. This is concurrency control.

// A3.4.4 · Distributed DB Features

Distributed Database Features

①
Concurrency Control — manages simultaneous access to shared data across nodes. Uses locking (one transaction locks data while using it) or timestamping (transactions ordered by unique timestamps) to prevent conflicts.
②
Data Consistency — ensures all copies of data across distributed nodes are synchronized and reflect the most recent, correct state. Strong consistency models ensure updates are visible to all nodes immediately.
③
Data Partitioning — distributes data across multiple nodes or servers to improve performance and scalability. Allows the system to handle large data volumes and high transaction rates efficiently.
④
Data Replication — copies of data are maintained on multiple nodes. If one node fails, others provide the data. Improves availability and fault tolerance.
⑤
Data Security — encrypt stored data, define role-based access controls, use multifactor authentication, and monitor for threats. Security is more complex when data spans many nodes and networks.

// A3.4.2 + A3.4.3 · Business Intelligence

The Business Intelligence Pipeline

Business intelligence (BI) is the process of turning raw data into actionable insights for decision-making.

RAW
Operational Databases — daily transactions stored in relational/NoSQL databases (OLTP)
ETL
Extract → Transform → Load — clean, format, and move data into the warehouse
DW
Data Warehouse — centralized, historical repository optimized for analysis
OLAP
Online Analytical Processing — multidimensional queries, slicing and dicing data
MINE
Data Mining — discover hidden patterns, build predictive models
BI
Reports + Dashboards + Decisions — insights presented to business leaders
// Common Errors · A3.4

Common Mistakes

✗
Confusing NoSQL with "no SQL at all" — many NoSQL databases have their own query languages. "NoSQL" means "not only SQL", not "no querying".
✗
Confusing data warehouse with operational database — a data warehouse is for historical analytics, not for daily transactions. You don't INSERT customer orders into a data warehouse in real time.
✗
Thinking OLAP and data mining are the same — OLAP explores predefined aggregated views; data mining discovers new patterns from raw data using algorithms.
✗
Forgetting that distributed ≠ replicated — partitioning splits data across nodes (each node has different data); replication copies data (each node has the same data). Systems often use both.
// Exam Strategy · A3.4

Key Exam Points

①
NoSQL type selection: Know which NoSQL type fits a given scenario. Social network = graph; user sessions = key-value; product catalog = document; IoT sensor data = wide-column.
②
Data warehouse characteristics: Remember the 5 key features — append-only, subject-oriented, integrated, time-variant, non-volatile. Be ready to explain each with an example.
③
ETL: Explain all three stages. State that extraction must not impact source systems, and data must be validated/cleaned before loading.
④
Distributed database features: For each feature (concurrency, consistency, partitioning, replication, security), be ready to explain why it matters and give a real-world example of failure without it.
⑤
OLAP vs Data Mining: OLAP = structured exploration of aggregated data; Data Mining = discovering unexpected patterns using algorithms. Both serve business intelligence but in different ways.
